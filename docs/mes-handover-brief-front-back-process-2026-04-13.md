# 鑫泰铝业统计系统 × MES 对接说明（前后工序分层版）

> 版本：2026-04-13  
> 发送对象：MES 厂商 / MES 对接方 / 现场实施方  
> 目的：明确本次统计系统与 MES 的边界、对接方式、字段需求、待确认问题，减少来回沟通成本。

---

## 1. 本次对接要解决什么

本次对接不是要让统计系统替代 MES，也不是要重做 MES。

本次要解决的是：

1. **前工序**（铸锭、热轧、铸轧）目前因为现场环境恶劣，**没有走 MES**  
   这部分需要由统计系统承担正式采集入口。

2. **后工序**继续以 **MES 为正式真源**  
   统计系统不改 MES 主流程，而是通过共享二维码 / MES 后续码去读取 MES 中的关键字段。

3. 管理层需要一套统一视图，能看到：
   - 当前生产推进情况
   - 合同推进情况
   - 在制状态
   - 车间状态
   - 经营估算相关视图

一句话：

> 前工序由统计系统采集，后工序由 MES 提供真相，统计系统负责把两段数据组织成统一的管理视图。

---

## 2. 当前已确认的业务边界

### 2.1 前工序边界

- 前工序范围：
  - 铸锭
  - 热轧
  - 铸轧
- 当前事实：
  - 这部分没有走 MES
  - 统计系统需要承担正式采集入口
- 前工序产出的正式对象：
  - **坯料**

### 2.2 后工序边界

- 后工序继续走 MES
- 后工序现场实际扫码：
  - 扫的是 **MES 自己的后续码**
- 后工序真源：
  - **MES**

### 2.3 前后关系

当前已确认：

- 前工序对象是 **坯料**
- 前后关系是：
  - **一坯对应一后续对象**

也就是说：

- 统计系统前段记录的是“坯料”
- MES 后段记录的是“后续对象”
- 中间需要有一条明确映射关系

---

## 3. 本次最关键的系统设计原则

### 原则 1：前后工序不是同一个对象连续贯穿

前工序记录的是 **坯料对象**，后工序记录的是 **MES 后续对象**。  
因此不能简单假设“前后都用一个码贯穿到底”。

### 原则 2：前工序真相和后工序真相分层

- 前工序真相：统计系统
- 后工序真相：MES

### 原则 3：需要建立“坯料 -> 后续对象”的映射

这条映射是本次对接的核心。  
如果这条映射不明确，后续所有：

- 合同推进
- 在制料追踪
- 车间状态汇总
- 经营估算

都会出现断层。

---

## 4. 对 MES 方的核心诉求

### 4.1 我们不要求 MES 改前工序流程

因为前工序本来就没有走 MES。

### 4.2 我们要求 MES 在后工序侧提供可读取事实

至少要能让统计系统通过 MES 后续码读取以下信息：

- 合金状态
- 生产工艺
- 当前所在工艺
- 规格
- 合同号（如有）
- 对应车间 / 机列 / 班次（如有）
- 状态
- 更新时间

### 4.3 我们最希望 MES 额外提供“来源坯料线索”

如果 MES 能提供以下任意一个，整个前后工序贯通就会非常顺：

1. 对应前工序坯料号
2. 来源坯料标识
3. 前工序随行信息
4. 任意可稳定反查到坯料对象的字段

如果 MES 能直接提供这条信息，统计系统就能自动完成“坯料 -> 后续对象”的绑定。

---

## 5. 本次建议的对接模式

### 5.1 前工序

由统计系统记录：

- 坯料标识
- 生产工序
- 当前状态
- 关键生产事实
- 采集时间
- 操作人 / 机列 / 班次（如有）

### 5.2 后工序

后工序现场继续扫 MES 码。  
统计系统不替代后工序 MES 录入，只做读取和关联。

### 5.3 映射建立时点

建议在：

> **进入后工序扫码时**

建立“前工序坯料对象 -> MES 后续对象”的映射。

这样原因是：

- 前工序对象已经存在
- 后工序对象也第一次正式进入 MES 真相链
- 这是前后工序第一次真正接轨的时间点

## 6. 这次最需要 MES 方明确回答的问题

请 MES 方按下面问题逐条确认。

### Q1. MES 后续码能否直接查询对象详情？

我们需要确认：

- 是否存在按后续码查询的 API
- 返回字段有哪些
- 是否支持实时查询

### Q2. MES 返回里是否包含“来源坯料信息”？

这是最关键的一条。

请确认 MES 是否能返回以下任意一种：

- 来源坯料号
- 来源坯料二维码
- 来源批次号
- 来源工序对象 ID
- 其他可稳定映射到前工序坯料对象的字段

### Q3. 如果 MES 当前没有来源坯料字段，是否有办法补充？

请确认可选方案：

1. 后工序对象新增一个“来源坯料标识”字段
2. 在 MES 接口中追加来源字段
3. 通过中间映射表提供查询接口
4. 其他等价方案

### Q4. 后工序对象详情接口的最小字段清单

请明确能否返回以下字段：

- `mes_code`
- `status`
- `alloy_status`
- `process_route`
- `current_process`
- `spec`
- `contract_no`
- `workshop_code`
- `machine_code`
- `shift_code`
- `event_time`
- `updated_at`

### Q5. 接口方式

请确认：

- 是 REST API 还是其他方式
- 是否有测试环境
- 是否有认证
- 是否有限流
- 是否支持增量拉取

## 7. 推荐给 MES 方的最小接口

如果 MES 方愿意配合，最推荐提供下面两类接口。

### 接口 A：按 MES 后续码查询详情

**请求示例**

```http
GET /mes-objects/{mes_code}
```

**期望返回示例**

```json
{
  "mes_code": "MES-20260413-0001",
  "status": "in_progress",
  "alloy_status": "A3003",
  "process_route": "cold_rolling",
  "current_process": "annealing",
  "spec": "0.8*1250*C",
  "contract_no": "HT-2026-001",
  "workshop_code": "AN01",
  "machine_code": "AN-03",
  "shift_code": "A",
  "event_time": "2026-04-13T10:00:00+08:00",
  "updated_at": "2026-04-13T10:00:10+08:00",
  "source_billet_no": "BL-20260413-0088"
}
```

其中最关键的是：

- `mes_code`
- `current_process`
- `status`
- `updated_at`
- `source_billet_no`（如能提供）

### 接口 B：按时间窗口拉取后工序对象变化

**请求示例**

```http
GET /mes-objects?updated_after=2026-04-13T10:00:00+08:00&limit=200
```

**用途**

- 统计系统做准实时同步
- 管理驾驶舱 5 分钟内更新

## 8. 如果 MES 现在无法提供“来源坯料字段”，我们需要提前知道

如果 MES 当前无法返回 `source_billet_no` 或等价字段，请直接明确告诉我们。

因为这会导致统计系统需要额外设计以下之一：

1. 人工确认映射
2. 固定业务规则自动映射
3. 中间映射台账

这三种方案都会增加复杂度。  
所以请优先确认：

> MES 是否能直接提供“后续对象对应的来源坯料信息”。

## 9. 当前我方系统状态（供 MES 方理解）

当前统计系统的定位不是替代 MES，而是：

- 前工序真实采集系统
- 后工序 MES 下游读取系统
- 统一的管理层经营估算与驾驶舱系统

因此我们和 MES 的关系是：

- **MES 管后工序生产真相**
- **统计系统管前工序采集和管理层视图**

## 11. 一句话总结

这次对接的核心不是“怎么把字段拉过来”，而是：

> **MES 是否能帮助我们把“前工序坯料对象”与“后工序 MES 对象”稳定接起来。**

如果这条线打通，后续统计、合同推进、在制料和经营驾驶舱都会顺。  
如果这条线打不通，后续所有看板都会存在前后断层。
