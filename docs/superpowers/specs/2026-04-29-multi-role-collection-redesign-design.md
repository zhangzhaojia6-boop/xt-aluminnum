# 多角色一线采集端重构设计

**日期：** 2026-04-29
**目标：** 将填报端从班长汇总模式重构为多角色一线采集模式，让各车间各项 owner 直接录入，消灭车间统计中间层。

## 背景

当前填报端以班长作为第一采集端，但实际一线数据来源分散在多个角色：
- 各机列主操按卷采集产量
- 各车间电工按班次采集能耗
- 机修工、液压工按班次采集耗材
- 全厂级 owner（水电气、成品库、计划科）按天采集全厂数据
- 质检按卷采集质检结论
- 各车间还有特有辅材需要耗材统计员采集

项目已有的 `workshop_templates.py` 和 `field_permissions.py` 定义了完整的角色字段权限体系，parse3 设计了一岗一表方向。本次设计将这些落地为可测试的多角色采集系统。

## 已确认决策

| 决策项 | 结论 |
|--------|------|
| 主操采集粒度 | 按卷录入，系统自动按班次汇总 |
| 辅助角色采集粒度 | 按车间+班次汇总录入 |
| 账号模式 | 一机一账号三班共用，系统按时间自动判班次，主操录入时填姓名 |
| 辅助角色登录 | 账号密码为主，车间QR码为快捷入口 |
| 前工序处理 | 前后工序统一按卷录入，前工序额外字段（铸速、炉温等）暂不做 |
| MES对接 | 暂不对接，后续拿有数据权限的MES账号写同步脚本 |
| 上线优先级 | 全角色同时上，今天让各车间owner开始测试 |

## 一、账号体系与登录流程

### 1.1 账号类型

**车间级角色：**

| 角色 | 系统代码 | 账号规则 | 登录方式 | 数据范围 |
|------|---------|---------|---------|---------|
| 主操 | shift_leader / OP | 机台绑定账号（已有） | 扫机台QR码自动登录 | 单机台 |
| 电工 | energy_stat / EN | `{workshop}-EN` | 账号密码 或 扫车间QR码 | 单车间 |
| 机修 | maintenance_lead / MT | `{workshop}-MT` | 账号密码 或 扫车间QR码 | 单车间 |
| 液压 | hydraulic_lead / HY | `{workshop}-HY` | 账号密码 或 扫车间QR码 | 单车间 |
| 耗材统计员 | consumable_stat / CS | `{workshop}-CS` | 账号密码 或 扫车间QR码 | 单车间 |
| 质检 | qc / QC | `{workshop}-QC` | 账号密码 或 扫车间QR码 | 单车间 |
| 称重员 | weigher / WG | `{workshop}-WG` | 账号密码 或 扫车间QR码 | 单车间 |

**全厂级角色：**

| 角色 | 系统代码 | 账号规则 | 登录方式 | 数据范围 |
|------|---------|---------|---------|---------|
| 水电气负责人 | utility_manager | `FACTORY-UM` | 账号密码 | 全厂 |
| 成品库负责人 | inventory_keeper | `FACTORY-IK` | 账号密码 | 全厂 |
| 计划科 | contracts | `FACTORY-CT` | 账号密码 | 全厂 |

### 1.2 登录流程

**主操**：扫机台QR码（`XT-{workshop_code}-{equipment_code}`）→ 自动登录 → 系统按当前时间判定班次 → 进入按卷录入页面。

**车间辅助角色**：账号密码登录 → 系统根据 user.role 路由到对应录入页面。或：扫车间QR码（`XT-{workshop_code}-WS`）→ 跳转登录页并带入车间参数 → 输入账号密码 → 进入该车间的补录页面。

**全厂级 owner**：账号密码登录 → 系统根据角色路由到对应的全厂日报补录页面。

### 1.3 班次自动判定

所有角色统一：根据当前时间 + ShiftConfig 自动确定当前班次。主操录入时填写操作人姓名（自由文本），用于追溯。

## 二、主操按卷录入流程

### 2.1 页面结构

主操扫码登录后进入**卷录入工作台**（新页面），包含：

1. **顶部状态栏**：机台名称、当前班次、业务日期、操作人姓名（首次填写后记住）
2. **卷列表**：本班次已录入的卷，显示卷号/合金/规格/投入/产出/状态
3. **底部操作按钮**："录入新卷" + "班次汇总预览"

### 2.2 单卷录入表单

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 坯料卷号 | 文本 | 是 | 手工输入或后续扫码带入 |
| 合金牌号 | 选择 | 是 | 下拉，常用合金列表 |
| 输入规格 | 文本 | 否 | 厚×宽 |
| 输出规格 | 文本 | 否 | |
| 上机时间 | 时间 | 否 | 默认当前时间 |
| 下机时间 | 时间 | 否 | |
| 投入量(kg) | 数字 | 是 | |
| 产出量(kg) | 数字 | 是 | |
| 废料量(kg) | 数字 | 否 | 自动建议值 = 投入 - 产出 |
| 备注 | 文本 | 否 | |

### 2.3 数据模型

写入现有 **WorkOrderEntry** 表。前后工序统一使用同一套录入表单和数据模型。

### 2.4 班次汇总预览

展示本班次所有卷的自动汇总：总投入、总产出、总废料、成品率、卷数。汇总数据自动同步到 ShiftProductionData。

## 三、辅助角色班次补录流程

### 3.1 各角色表单字段

**电工 (energy_stat)**：电耗(kWh)、气耗(m³)。铸轧车间额外：按合金系分气耗（1系/3系/5系）。

**机修 (maintenance_lead)**：停机分钟(min)、停机原因。

**液压 (hydraulic_lead)**：液压油32#(桶)、液压油46#(桶)、齿轮油(桶)。

**耗材统计员 (consumable_stat)**，按车间类型动态渲染：
- casting（铸轧）：液化气吨耗(kg)、钛丝吨耗(kg)、钢带吨耗(kg)、镁锭吨耗(kg)、锰剂吨耗(kg)、铁剂吨耗(kg)、铜剂吨耗(kg)
- hot_roll（热轧）：热轧乳液吨耗(kg)
- cold_roll（冷轧）：轧制油吨耗(kg)、飞滤剂吨耗(kg)、硅藻土吨耗(kg)、白土吨耗(kg)、滤布日用(米)、高温胶带日用(卷)、再生油出(kg)、再生油回(kg)
- finishing（精整/拉矫）：轧制油吨耗(kg)、D40吨耗(kg)、钢板吨耗(kg)、钢带吨耗(kg)、钢带扣吨耗(kg)、高温胶带日用(卷)

**质检 (qc)**：按卷录入质检结论、质检备注、质检附加信息。写入 WorkOrderEntry.qc_* 字段。

**称重员 (weigher)**：按卷核实投入量、核实产出量。写入 WorkOrderEntry.verified_* 字段。

### 3.2 数据写入

车间级辅助角色（电工/机修/液压/耗材统计员）写入 **MobileShiftReport**，多角色写同一条记录的不同字段，通过 workshop_templates 的 role_write 隔离。耗材统计员的动态字段存入 extra_payload（JSON）。

质检和称重员按卷写入 **WorkOrderEntry** 的对应字段。

### 3.3 车间QR码

新增车间级QR码，格式：`XT-{workshop_code}-WS`。扫码后跳转移动端登录页并自动带入车间参数。

## 四、全厂级 Owner 录入

### 4.1 水电气负责人 (utility_manager)

页面分三段：用电 → 天然气 → 用水。

字段（已在 workshop_templates UTILITY_OWNER_FIELDS 中定义）：
- 用电：全厂用电(kWh)、新厂用电(kWh)、园区用电(kWh)
- 天然气：铸轧用气(m³)、熔炼炉用气(m³)、加热炉用气(m³)、锅炉用气(m³)、天然气总量(m³)
- 用水：地下水(吨)、自来水(吨)

### 4.2 成品库负责人 (inventory_keeper)

页面分三段：今日入库 → 今日发货 → 结存与备料。

字段（已在 workshop_templates INVENTORY_OWNER_FIELDS 中定义）：
- 入库：入库成品(吨)、入库面积(㎡)、本厂入园区(吨)、园区入成品库(吨)、月累计入库(吨)、月累计入库面积(㎡)
- 发货：对外发货(吨)、发货面积(㎡)、月累计发货(吨)、月累计发货面积(㎡)
- 结存：寄存吨位(吨)、成品库存(吨)、剪切备料(吨)

### 4.3 计划科 (contracts)

页面分三段：当日合同 → 月累计与余合同 → 投料与坯料。

字段（已在 workshop_templates CONTRACT_PROGRESS_FIELDS 中定义）：
- 当日：当日接合同(吨)、当日热轧合同(吨)
- 月累计：月累计合同(吨)、月累计热轧合同(吨)、余合同量(吨)、余热轧合同(吨)、余合同较昨日(吨)
- 投料：坯料总量(吨)、当日投料(吨)、月累计投料(吨)

### 4.4 数据写入

全厂级 owner 继续写入 WorkOrderEntry.extra_payload（与 parse3 设计一致），通过 source precedence 识别 owner-only 数据。

## 五、数据汇总与自动报表

### 5.1 汇总链

```
一线采集层（按卷）
├── 主操 shift_leader → WorkOrderEntry（产量）
├── 质检 qc → WorkOrderEntry.qc_*（质检结论）
└── 称重员 weigher → WorkOrderEntry.verified_*（核实重量）

车间采集层（按车间×班次）
├── 电工 energy_stat → MobileShiftReport（电耗/气耗）
├── 机修 maintenance_lead → MobileShiftReport（停机）
├── 液压 hydraulic_lead → MobileShiftReport（液压油/齿轮油）
└── 耗材统计员 consumable_stat → MobileShiftReport.extra_payload（辅材）

全厂采集层（按天）
├── 水电气 utility_manager → WorkOrderEntry.extra_payload（全厂水电气）
├── 成品库 inventory_keeper → WorkOrderEntry.extra_payload（出入库/库存）
└── 计划科 contracts → WorkOrderEntry.extra_payload（合同/投料进度）

    ↓ aggregator-agent 自动汇总

自动报表层
├── 综合日报表 → 替代"鑫泰每日产量"Excel
├── 能耗月报 → 替代"各车间能耗统计表"
├── 天然气月报 → 替代"各车间天然气用量统计表"
├── 耗材日报 → 替代"耗材表"
├── 成品率月报 → 替代"各车间成品率"
└── 合同进度报表 → 替代"合同报表"

    ↓ reporter-agent 自动推送

管理层（factory-observer / workshop-observer）
```

### 5.2 自动报表对标

| 自动报表 | 替代的手工报表 | 数据来源 |
|---------|-------------|---------|
| 综合日报表 | 鑫泰每日产量.xls | 主操产量 + 电工能耗 + 成品库出入库 + 计划科合同 |
| 能耗月报 | 各车间能耗统计表.xls | 电工按班次录入 → 按天×车间汇总 |
| 天然气月报 | 各车间天然气用量统计表.xls | 电工气耗 + 水电气负责人全厂气耗 |
| 耗材日报 | 耗材表.xls | 电工能耗 + 液压耗油 + 耗材统计员辅材 |
| 成品率月报 | 各车间成品率.xlsx | 主操投入/产出 → 系统自动计算 |
| 合同进度报表 | 合同报表.xlsx | 计划科录入 |

### 5.3 汇总触发

- 主操提交一卷 → 实时更新该机台×班次汇总
- 辅助角色提交 → 实时更新该车间×班次对应字段
- 班次结束 → validator-agent 自动校验，锁定班次数据
- 每日定时 → aggregator-agent 生成日报 → reporter-agent 推送

## 七、MES 对接备忘

MES 系统（mes.xintaily.com）探索发现：
- ASP.NET MVC + SQL Server（端口1433开放）
- 铸轧车间看板有9条机列（1#~9#），分布在铸一到铸六+铸锭+热轧
- 前工序产出也是按**坯料卷号**跟踪，前后工序通过坯料卷号串联
- 当前可用账号（wxn/admin）数据权限受限，需要拿有数据权限的账号
- Web API 可通过 curl 模拟登录调用（已验证），后续可写定时同步脚本
