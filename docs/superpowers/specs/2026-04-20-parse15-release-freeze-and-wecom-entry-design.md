# Parse 15 Release Freeze And WeCom Entry Design

## 背景

parse14 已经把 `analysis_handoff.action_matrix` 收成稳定 contract，当前产品主链也已经完成一轮严格使用性收口。

现在继续扩功能的收益已经低于“先冻结发布基线、准备生产入口、跑真实 UAT”的收益。

当前真正阻塞内部试点上云的，已经不是主链代码，而是：

1. 发布边界是否冻结
2. 生产环境变量是否准备好
3. 企业微信生产入口是否准备好
4. 现场 UAT 是否可执行

## parse15 目标

把下一阶段切成“发布闸门”而不是“继续堆功能”：

1. 冻结当前可发布范围
2. 固化企业微信生产入口准备项
3. 固化现场 UAT 执行项
4. 明确哪些是代码已准备、哪些是外部阻塞

## 设计原则

- 不扩新业务范围
- 不引入新依赖
- 不假装已经上云
- 只做系统侧可确定的准备和闸门梳理
- 把需要用户现场提供的真实输入明确列出来

## 本阶段不做

- 不做 MES 联调
- 不做成本/排产/经营自动化
- 不做没有凭据的真实云端操作
- 不伪造企业微信后台配置完成状态

## 验收标准

- 发布冻结基线有清晰边界
- 企业微信生产入口准备项可直接执行
- UAT 清单可直接拿去现场跑
- 明确列出外部阻塞，不把外部前置条件伪装成已完成
