# Phase 2 Internal Dual-Surface Motion Design

## 背景

Phase 1 已经把系统主线收口成“岗位直录 -> 系统自动校验/汇总 -> 领导直达”，但当前前端还有两个明显问题：

1. 填报端和审阅端混在同一套表达里，填报人会看到过多与自己无关的信息
2. 审阅端虽然开始体现智能体联动，但还不够像“真实流程追踪”，也不够一眼看懂

这一轮不做对外官网，先把内部系统打磨成两个完全独立的端：

- 填报端：只服务主操与专项 owner
- 审阅端：只服务管理员与管理层

同时把后端底层核查列为最高优先级，确保界面呈现出的“自动链路”和真实数据链一致，不是视觉包装。

## 已确认决策

- Phase 1 继续不接 MES；保留接口缝，但前端主路径不显式展示 MES
- 现有统计角色完全退出主流程，不作为系统主叙事
- 主操/机台账号直接报，专项 owner 同期纳入
- 填报端与审阅端是两个独立窗口，权限和体验完全分离
- 填报端优先级高于审阅端，先保证更快、更稳、更顺手
- 审阅端必须做到“三岁小孩也能看懂数据从哪来、现在到哪了、哪里有问题”
- AI 感主要体现在接力结构、处理结果和流程追踪，不靠大段说明文案
- 视觉方向参考 Stripe Dashboard 的清爽、克制、科技感，不做字面克隆
- 机器人视觉采用“软萌泡泡”方向，用来表达前场/后场流程追踪

## 设计

### 1. 双端信息架构

#### 填报端

- 单独入口、单独路由、单独视觉语言
- 非管理员角色默认只能进入填报端
- 页面只保留：
  - 今日任务
  - 当前页输入
  - 退回原因
  - 提交结果
- 页面移除：
  - 智能体小队说明
  - 接力轨道
  - 过量 facts 区
  - 大段“模块负责什么”的解释文案

#### 审阅端

- 单独入口、单独路由、单独壳层
- 管理员进入后先看“今日流转是否顺畅”，再看细节
- 首页优先顺序固定为：
  - 数据从哪来
  - 现在流到哪
  - 今日有无缺报/异常/退回/交付风险
  - 已汇总出的关键结果
  - 预留给后端 AI agent 的分析/预测/核算挂点

### 2. 填报端交互设计

- 填报端采用“模块分页滑动”而不是长表单滚动
- 每个模块是一整屏，左右滑动切换，像手机桌面切屏
- 交互目标：
  - 轻微阻尼
  - 吸附到整页
  - 滑动不掉帧
  - 输入过程中不误触切页
- 分页按业务模块组织，不按单字段碎片化切分
- 同一岗位默认 3 到 5 个模块以内完成，不追求把所有字段塞在一屏
- 每屏只保留少量主字段，用图标、数字块、状态块辅助理解，减少说明文字
- 提交反馈要明显，但短，不使用冗长解释

### 3. 审阅端流程追踪设计

- 审阅端的 AI 互动感不做聊天式，不做“AI 说明卡片刷屏”
- 改为“前场小机器人 + 后场小机器人”的流程追踪动画
- 目标效果：
  - 看得见数据由谁提交
  - 看得见哪一只机器人正在接数、校验、催报、汇总、预警
  - 看得见当前卡在哪一步
  - 看得见结果已经推送到哪里
- 机器人承担的是“流程状态表达”，不是功能入口
- 机器人风格采用软萌泡泡造型，动作轻、弹性弱、不夸张
- 动画重点是数据脉冲、轻微漂浮、接力传输、状态亮起，不做大型炫技动画

### 4. 审阅端内容层级

- 第一层：来源图层
  - 主操直录
  - 专项 owner 补录
  - 系统自动汇总
- 第二层：运行图层
  - 缺报
  - 异常
  - 退回
  - 交付风险
- 第三层：结果图层
  - 当日产量
  - 月累计
  - 库存/发货
  - 合同/能耗等已接入的关键结果
- 第四层：未来能力挂点
  - 分析
  - 预测
  - 核算
  - 建议

四层都优先用图形、状态和极短标签表达，尽量少字段、少段落。

### 5. 全局视觉与动效原则

- 配色改为冷白、深蓝、蓝青的 Stripe 式清爽科技感
- 减少厚重渐变、过暖色、过多说明框
- 强化：
  - 轻微悬浮
  - 掠光
  - 状态脉冲
  - 数据流线
  - 分页滑动阻尼
- 保持克制：
  - 不做到处乱飞的粒子
  - 不做大面积霓虹
  - 不做强 AI 宣传语
- 其他页面也逐步统一到这个动效语言，但填报端优先保证性能和顺手

### 6. 后端核查优先级

这一轮 UI 改造必须建立在后端核查之上，核查顺序固定为：

1. 数据口径与汇总正确性
2. 零人工自动链路是否成立
3. 权限隔离是否与双端分离一致

重点不是“接口还能返回”，而是确认以下事实：

- 核心指标都能追溯到真实提交来源
- happy path 不依赖 reviewer 或人工确认
- 主操和专项 owner 不会看到审阅端信息
- 审阅端展示的数据流不会越权泄露不该看的细节
- 审阅端里机器人表达的每一个阶段，都有真实 service / workflow 对应

### 7. 本轮不做

- 不做对外展示首页
- 不做 MES 接入
- 不做排产、成本、合同排序自动化的主流程实现
- 不做新的重后台统计工作台
- 不为了视觉效果重建人工审核链

## 验收标准

- 填报人进入后只看到填报端，且能通过滑屏模块快速完成录入
- 填报端显著减字，去掉多余的“AI 说明型字段”和职责解释块
- 审阅端能一眼看懂“数据来源 -> 当前流转 -> 今日风险 -> 当前结果”
- 审阅端的 AI 互动感来自流程追踪动画，而不是长文案
- 后端关键链路有测试或核查证据支撑，不靠口头判断
- 后续可以在此基础上进入 parse2，而不需要重新推翻信息架构

## 下一步

1. 先做后端核查清单与测试补强
2. 再拆分填报端/审阅端路由与权限入口
3. 然后做填报端滑屏工作台
4. 最后做审阅端流程追踪动画、视觉统一和浏览器验收
