# 管理端一屏态势 + 填报端字段收敛设计

日期：2026-05-02
状态：待用户审阅
适用系统：鑫泰铝业 数据中枢

## 背景

当前系统已经具备移动填报、实时聚合、日报交付、质量异常、主数据配置等能力，但管理端入口和信息层级过多。管理者进入后台后，需要在多个页面之间寻找“哪些机列填了、哪些没填、哪里异常、今天工厂状态是否可信”。这导致系统看起来复杂、文字多、不够直观。

现有代码中有两条可复用基础：

- `GET /api/v1/aggregation/live` 已能返回车间、机台、班次、提交状态、投入产出和考勤状态。
- `frontend/src/views/reports/LiveDashboard.vue` 已有“车间 × 机列 × 班次”的状态矩阵，但目前归在系统运维/设置入口下，命名和视觉重点不适合作为管理端主屏。

本设计目标不是继续新增功能，而是把现有能力收敛成一张能看懂的工厂状态板，并把冗余入口降级或隐藏。

## 目标

1. 管理端首屏在 10 秒内回答四个问题：
   - 哪些机列/班次已经填报？
   - 哪些机列/班次缺报或异常？
   - 今日产量、产出、成材率、提交进度是多少？
   - 当前数据源是否可信，是否来自 MES 投影或工单兼容口径？
2. 填报端继续减负：不同车间、不同机列只看到本岗位必须填的字段。
3. 后端先收敛状态口径和测试，不做大规模拆服务。
4. 管理端视觉从“文字解释型后台”转成“工业态势看板”：少段落、多状态块、多数字、多颜色状态。
5. 冗余功能先从主导航降级，确认无生产依赖后再删除。

## 非目标

- 不重写后端架构。
- 不引入新的前端 UI 框架。
- 不新增第三方实时服务。
- 不把产品称为 MES；MES 只作为外部数据源或集成边界出现。
- 不删除仍有路由、测试或线上入口依赖的功能。

## 推荐方案

采用“薄聚合 + 强前端态势板”的方案：

- 后端继续使用 `realtime_service.build_live_aggregation` 作为状态数据来源，必要时补齐字段和测试。
- 前端把 `LiveDashboard` 的核心矩阵提升为管理端工厂状态主屏，重命名和重排为“工厂实时态势”。
- `FactoryDirector` 只保留厂长关心的摘要和跳转，不再承担完整实时矩阵。
- 导航保留高频入口，低频配置入口放入管理员分组。
- 填报端字段继续由模板驱动，不在页面里硬编码车间差异。

这个方案利用已有接口和页面，不会把已上线链路打碎，风险最低。

## 信息架构

管理端主导航收敛为六类：

1. 总览：管理者默认入口，展示全厂实时态势。
2. 工厂状态：车间、机列、班次填报矩阵。
3. 填报审核：待审、退回、缺报补录。
4. 日报交付：日报、导出、交付状态。
5. 异常质量：质量、差异、异常处置。
6. 主数据：车间、机列、模板、用户、权限。

降级入口：

- AI 工作台、成本核算、治理、导入历史、系统设置不出现在普通管理者第一层导航。
- 管理员仍可访问这些页面，但入口放在二级管理分组。
- 参考原型页和占位页不应作为生产导航项出现。

## 首屏设计

首屏分三层。

第一层：全厂状态条

- 今日产出
- 提交进度
- 缺报单元数
- 异常/退回数
- 数据源状态
- 最后更新时间

第二层：车间状态矩阵

- 行：车间，展开后为机列。
- 列：班次。
- 单元格：填报状态。
- 状态颜色：
  - 绿色：已提交或已确认
  - 黄色：进行中
  - 红色：缺报、退回、异常
  - 灰色：不适用

第三层：点击单元格抽屉

- 显示该车间、机列、班次的卷明细。
- 显示投入、产出、废料、成材率。
- 显示填报状态和数据源。
- 有异常时只显示关键原因，不堆长说明。

## 数据流

```text
移动填报 / MES 投影 / 考勤确认
        |
        v
后端实时聚合服务
        |
        v
GET /api/v1/aggregation/live
        |
        v
管理端工厂实时态势
        |
        v
单元格详情 GET /api/v1/aggregation/live/detail
```

前端不自行推断核心状态。填报状态、适用性、数据源、聚合数字由后端给出，前端只负责展示、筛选和交互。

## 后端状态口径

保留现有状态字段，并将其契约化：

- `submission_status`
  - `not_started`
  - `in_progress`
  - `all_submitted`
  - `not_applicable`
- `attendance_status`
  - `not_started`
  - `pending`
  - `confirmed`
  - `not_applicable`
- `data_source`
  - `mes_projection`
  - `work_order_compat`
- `overall_progress`
  - `submitted_cells`
  - `total_cells`

需要补齐的口径：

- 缺报单元数：所有适用单元格中 `not_started` 的数量。
- 异常单元数：退回、异常考勤、异常质量相关状态的数量。
- 响应性状态：接口延迟、SSE 连接状态、最后更新时间只作为页面顶部健康信息，不混入生产指标。

## 填报端字段收敛

填报端继续走模板驱动：

- 每个车间模板只暴露本车间必须填字段。
- 每个机列可通过模板或设备属性进一步限制字段。
- 合金、状态、规格等字段使用下拉或结构化输入。
- 废料、成材率等计算字段不在主操录入页展示。
- 备注字段保留，但不作为必填。

移动端首屏只保留：

- 当前车间、机列、班次
- 已录卷数
- 录一卷入口
- 本班汇总入口
- 异常补录入口

## 前端改造范围

主要文件：

- `frontend/src/views/reports/LiveDashboard.vue`
  - 改造成“工厂实时态势”页面。
  - 去掉运维口吻标题和长说明。
  - 强化矩阵视觉状态和顶部指标条。
- `frontend/src/views/dashboard/FactoryDirector.vue`
  - 删除 `display:none` 的旧战争室、旧 ribbon、旧 hero grid。
  - 只保留摘要卡和进入实时态势的入口。
- `frontend/src/config/manage-navigation.js`
  - 收敛导航分组。
  - 普通管理者不直接暴露低频配置入口。
- `frontend/src/router/index.js`
  - 调整 `/manage/overview` 或 `/manage/factory` 的默认落点。
  - 保留旧路径重定向，避免线上链接失效。
- `frontend/src/views/mobile/CoilEntryWorkbench.vue`
  - 按车间和机列继续减少字段。
  - 增强数字输入校验和错误提示。

## 后端改造范围

主要文件：

- `backend/app/services/realtime_service.py`
  - 补齐缺报数、异常数、数据源状态。
  - 保持现有聚合函数边界，不新增平行聚合服务。
- `backend/app/schemas/realtime.py`
  - 明确输出模型字段。
- `backend/app/routers/realtime.py`
  - 保持现有路由，必要时补 query 参数校验。
- `backend/tests/test_realtime_routes.py`
  - 覆盖接口契约。
- `backend/tests/test_workshop_reporting_status.py`
  - 保持车间缺报状态回归。

## 冗余清理规则

第一阶段只隐藏，不删除：

- 零引用页面。
- 参考原型页。
- 占位页。
- 低频配置入口。

第二阶段再删除：

- 已无路由入口。
- 无测试依赖。
- 无文档宣称为生产能力。
- 线上用户不再访问。

删除前必须用文件引用、路由表、测试和构建验证四项确认。

## 测试策略

后端：

- `test_live_aggregation_endpoint_calls_service`
- 聚合状态单测：缺报、进行中、已完成、不适用。
- 权限单测：车间用户只能看到本车间。
- 详情接口单测：机列、班次、车间参数必须命中正确数据。
- 数据源回退单测：MES 无数据时回退工单兼容口径。

前端：

- 导航测试：普通管理者只看到收敛后的入口。
- 状态矩阵测试：四种状态颜色和文字正确。
- 抽屉详情测试：点击单元格能加载详情。
- 移动填报测试：字段按模板隐藏和展示。
- 构建验证：`npm --prefix frontend run build`。

人工验收：

- 桌面端 1440px 和 1920px 首屏无文字拥挤。
- 平板尺寸下矩阵可横向滚动，状态不变形。
- 手机端不进入管理矩阵，保持填报优先。

## 回滚

前端回滚成本低：

- 保留旧路由重定向。
- `LiveDashboard` 仍使用原接口。
- 导航改动可单文件回滚。

后端回滚成本中低：

- 只向聚合响应新增字段，不删除现有字段。
- 前端对新增字段做默认值兜底。
- 数据库结构不变，因此无需迁移回滚。

## 风险与应对

风险一：主数据中的车间、机列、班次关系不准确。
应对：实现前先跑主数据一致性检查，发现异常时在管理端顶部显示配置异常，不把错误状态伪装成缺报。

风险二：矩阵信息量过大。
应对：默认按车间折叠，异常车间自动置顶，机列详情按需展开。

风险三：管理端入口被收敛后用户找不到低频功能。
应对：管理员二级入口保留，旧 URL 重定向保留。

风险四：状态颜色表达不清。
应对：状态文字始终和颜色一起出现，不只依赖颜色。

## 验收标准

- 管理者进入管理端第一屏即可看到全厂提交进度、缺报数、异常数和数据源状态。
- 任一车间缺报时，对应车间和机列状态在矩阵中可见。
- 点击任一机列班次单元格，可看到该单元格卷明细。
- 普通管理导航不再出现大量低频入口。
- 填报端主操录一卷时不看到废料、成材率等计算字段。
- 后端和前端相关测试通过。

## 依赖

不需要新增外部服务、API key、第三方账号或新命令行工具。继续使用现有 FastAPI、Vue 3、Element Plus、npm、pytest。
