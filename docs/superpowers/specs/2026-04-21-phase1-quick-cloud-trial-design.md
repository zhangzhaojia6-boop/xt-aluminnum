# Phase 1 Quick Cloud Trial Design

## 背景

当前项目已经具备 `主操直录 + 专项 owner 补录 + 系统自动汇总 + 管理端审阅` 的核心骨架，也已经有一批 focused tests、build 和 E2E 用例资产。

用户当前明确希望采用更接近《精益创业》的方式推进：

1. 不再继续闭门扩功能
2. 先把系统放到云上，尽快让真实用户开始试跑
3. 用真实反馈驱动后续改进
4. 用户本身不熟悉代码、部署和数据库，希望部署与后续运维尽量被收成“低门槛操作”

## 本轮目标

把当前系统收口成一个可用于 `一个车间快速试跑` 的云上版本。

本轮目标不是“正式全量上线”，而是：

1. 上线一版可真实访问的 HTTPS 试跑环境
2. 覆盖一个车间的管理员、主操/机台账号、专项 owner
3. 保留后续快速迭代能力
4. 让上线后的运维操作尽量标准化和傻瓜化

## 已确认决策

### 上线路线

采用 `方案 1：快速试跑`。

即：

1. 先开放浏览器访问
2. 企业微信正式入口先不接
3. 先验证真实车间试跑链路

### 试跑范围

采用 `一个车间试跑`。

即：

1. 1 个管理员
2. 1 个车间的主操/机台账号
3. 该车间所需专项 owner

### 部署结构

采用 `单机快速试跑`。

即：

1. 一台 Linux 云主机
2. 应用与 PostgreSQL 都放在同一台机器
3. 使用 `Docker Compose` 管理
4. 域名已具备并可解析到该云主机

## GitHub 决策

### 是否要先连 GitHub

建议：`要，优先做，而且应在上云前完成。`

原因不是为了“更工程化”，而是为了试跑阶段的实际效率：

1. 云主机可以直接 `git pull`，后续迭代不用再手工传代码包
2. 每一轮试跑修复都能形成清晰版本基线
3. 需要回滚时更简单
4. 后续加 CI、做多人协作、做发布标签都更顺手

### 本轮对 GitHub 的要求

本轮不要求一开始就把 GitHub 流程做复杂，但至少要满足：

1. 仓库有远端
2. 云主机可以拉取代码
3. 生产 `.env`、证书、密钥不提交到 GitHub
4. 后续每次试跑发布都能对应到一个明确版本

### 本轮不要求

1. 不要求现在就做完整 GitHub Flow
2. 不要求现在就上 Actions/CD
3. 不要求现在就做自动部署

## 目标架构

### 云上服务

单台 Linux 云主机运行以下服务：

1. `nginx`
   - 对外暴露 80/443
   - 承接域名和 HTTPS
   - 转发 API 与前端静态资源
2. `backend`
   - 运行 FastAPI
   - 启动时执行数据库迁移与初始化脚本
3. `db`
   - PostgreSQL 15
   - 与应用同机部署

### 访问方式

用户首轮通过浏览器访问：

1. `/mobile`
2. `/review/factory`
3. `/review/workshop`

企业微信正式入口不属于本轮上线阻断项。

## 数据库策略

本轮数据库遵循“够用、可恢复、少操作”的原则。

### 采用方案

1. 继续使用 Compose 内置 PostgreSQL
2. 数据与应用同机部署
3. 通过初始化脚本生成管理员与主数据骨架
4. 每日做一次最小备份

### 运维要求

数据库运维不要求用户理解 SQL 或 PostgreSQL 内部细节。

本轮需要被产品化成固定动作：

1. 初始化
2. 备份
3. 恢复
4. 查看容器状态
5. 查看日志

### 本轮不做

1. 不做托管数据库
2. 不做读写分离
3. 不做高可用主从
4. 不做复杂数据迁移平台

## 发布策略

本轮发布方式遵循 `冻结一版 -> 上云 -> 小范围试跑 -> 快速改进`。

### Day 0：冻结基线

1. 清点当前工作区改动
2. 收出一版“试跑候选基线”
3. 明确本轮不再扩新业务范围

### Day 1：准备云端资产

1. 云主机
2. 域名解析
3. HTTPS 证书
4. 生产 `.env`
5. GitHub 远端与云主机拉取方式

### Day 2：起服务

1. 云主机拉代码
2. 放置 `.env`
3. 放置证书
4. Compose 启动
5. 验证 `healthz / readyz`

### Day 3：最小试跑

1. 管理员登录
2. 主操/机台账号提交
3. 专项 owner 补录
4. 审阅端确认来源、状态、结果变化

## 运维模式

本轮目标不是把用户训练成运维，而是把运维动作固化成固定命令或文档步骤。

### 需要被收口的运维动作

1. 一键部署
2. 一键更新
3. 一键看日志
4. 一键备份数据库
5. 一键回滚
6. 一键健康检查

### 运维边界

我可以帮助完成：

1. 部署流程设计
2. 生产 `.env` 模板收口
3. 数据库初始化路径
4. 健康检查
5. 备份/恢复脚本
6. 回滚设计
7. 上线检查清单

仍需要用户在云厂商侧完成的动作主要是：

1. 提供 SSH 登录方式
2. 域名解析配置
3. 证书申请或上传
4. 云平台账号级权限操作

## 成功标准

本轮“快速试跑成功”定义为：

1. 云上 HTTPS 可访问
2. 管理员能登录审阅端
3. 一个车间主操/机台账号能提交
4. 一个专项 owner 能补录
5. 审阅端能看到来源、状态和结果变化
6. `readyz` 返回 `hard_gate_passed=true`
7. 数据库可以备份
8. 服务异常后可通过标准步骤恢复

## 风险与取舍

### 接受的取舍

1. 单机部署不是最终形态
2. 数据库和应用同机的容灾能力有限
3. 企业微信正式入口先不接
4. 先服务一个车间，而不是全厂

### 不接受的风险

1. 生产 `.env` 和密钥误提交到 Git
2. 没有备份就上云
3. 没有健康检查就让现场试跑
4. 没有最小回滚路径就直接给真实用户使用

## 本轮不做

1. MES 联调
2. 企业微信正式消息打通
3. 托管数据库
4. 多环境复杂发布体系
5. 自动化 CI/CD
6. 多车间同时放量
7. 新一轮功能扩展

## 下一步

在该设计获批后，下一步进入 implementation plan，重点拆成：

1. 发布基线清点
2. GitHub 连接与云端拉取路径
3. 生产 `.env` 与证书准备
4. 单机 Compose 上云流程
5. 数据库初始化/备份/恢复
6. 健康检查与最小试跑验收
