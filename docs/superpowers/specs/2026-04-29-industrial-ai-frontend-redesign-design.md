# 鑫泰铝业 MES 工业 AI 前端重构设计

Date: 2026-04-29

## 1. Goal

把前端从普通后台样式升级为“鑫泰全厂作战地图 + AI 总管侧舱 + 自动执行轨”的工业 AI 管理系统。

本轮升级为 frontend-only 大面积视觉返工，不是局部修补。目标是让登录页、管理端、总览、AI 工作台、主要业务页面和移动端入口形成同一套高完成度工业 AI 产品气质。

本轮设计只定义前端视觉、交互和实施边界，不改变权限、排班、设备绑定、MES 同步、日报发布、审计日志、readyz、hard gate、AI 校验逻辑。

## 2. Visual Thesis

鑫泰铝业 MES 的第一屏应像一个真实工业 AI 总经理驾驶舱：中央看全厂，右侧做判断，底部展示系统正在自动执行的闭环。视觉要有铝业、熔铸、产线、批次流的品牌记忆点，同时保持企业生产系统需要的清晰、克制和可扫描。

质量基准采用仓库全局规则中的前端要求。用户原话“像opus 4.7在Claude code编码设计前端一样，参考其设计前端多好看，你gpt5.5设计的前端太难看了”是本轮的硬约束：必须避免通用 GPT 式模板 UI，追求 Opus 级的构图、品牌、交互细节和浏览器验证完成度。

这意味着每个改动都要回答一个问题：它是否让系统更像真实、昂贵、可用的工业 AI 产品，而不是默认后台皮肤。

## 3. Design Direction

已选方向：A + B。

- A: 全厂作战地图为主，AI 总管侧舱在右侧。
- B: 数字孪生产线抽象和执行轨增强工业感。
- D: 微交互组合，包含地图扫线、执行轨推进、问答转动作。

核心画面：

- 中央：全厂作战地图，抽象表达熔炉、机台、铸锭、批次流、异常点。
- 右侧：AI 总管侧舱，展示预测、判断、已执行动作、下一步方向。
- 底部：自动执行轨，固定表达“发现 → 判断 → 执行 → 留痕 → 发布”。
- 顶部：品牌化鑫泰区域，不再只有单字方块 logo。

## 4. AI Brain Product Role

AI Brain 不是普通聊天助手，而是“总管全场的职业经理人”。

它在界面中的职责：

- 预测：提前发现产线、填报、日报、同步风险。
- 分析：解释异常原因和影响范围。
- 给方向：给出下一步处理方向。
- 真正做事：在规则允许范围内自动催报、重新校验、生成摘要、触发发布或记录动作。
- 留痕：所有自动动作必须能被审计和追溯。

本设计不新增越权执行能力。前端只呈现已有或后续明确授权的动作，不绕过后端权限、审计或 workflow dispatcher。

## 5. AI Workstation Constraint

必须保留现有 AI 工作台的极简问答助手方式。

AI 工作台不是全部改造成复杂地图页。它应继续保持：

- 输入框清楚，问答路径短。
- 对话消息易读，像高质量生产工具，不像营销展示页。
- 支持 AI 辅助操作面板，把回答进一步转成可执行动作。

允许增强：

- 思考中状态增加更丰富的动效，例如呼吸式状态点、分段推理轨、工具调用闪烁、答案生成进度。
- 工具调用状态明确区分“分析中、查数据、生成动作、执行中、已完成”。
- 问答转动作时，动作卡片从回答下方自然展开，不打断主对话流。

不允许：

- 删除极简问答主路径。
- 把 AI 工作台改成只有地图或大屏。
- 用大段说明文字解释功能。
- 添加与业务无关的装饰动画。

## 6. Brand And Logo System

品牌系统需要从“鑫”字方块升级为一套可复用的图形语言。

图形元素：

- 铝锭：几何块体、倒角、冷白金属边。
- 熔炉：内核暖铜色，只作为小面积能量点。
- 批次流：细线、节点、箭头、轨道。
- AI 中枢：克制的环形扫描或聚合节点。

使用原则：

- Logo 图形可在登录页、管理端侧栏、移动端顶栏、AI Brain 页使用。
- 不使用复杂插画，不使用大面积渐变背景。
- 品牌色以冷白、石墨、工业蓝为主，小面积铜色代表熔铸能量。
- 图形必须在 28px、32px、44px 和登录页大尺寸下都可读。

## 7. Surface Plan

本轮是大面积视觉返工，但仍只在前端范围内推进。

第一批重构页面：

1. 登录页
   - 强化品牌第一印象。
   - 使用风格化 logo 和工厂作战地图的静态缩影。
   - 保持登录表单清楚，不做营销落地页。

2. 管理端 shell
   - 侧栏、顶栏、命令入口和品牌区统一。
   - 从普通导航升级为工业控制台框架。
   - 保持现有路由、权限和导航来源。

3. 管理端总览
   - 使用全厂作战地图作为第一屏核心。
   - 右侧加入 AI 总管侧舱。
   - 底部加入自动执行轨。

4. AI Brain / AI 工作台
   - 保留极简问答助手。
   - 增强思考中、工具调用、执行动作、完成状态。
   - 问答与动作面板并存，优先保证对话可读。

5. 移动端 /entry
   - 使用同一品牌系统的轻量版本。
   - 不引入复杂地图。
   - 继续保持大按钮、大字、一线工人低认知负担。

第二阶段再扩展：

- 各 `/manage` 模块页的卡片、表格、筛选器、空状态、批量操作条。
- Dashboard、导入、主数据、班次详情、质量、日报、差异、考勤等页面的视觉重铺。
- 移动端 `/entry`、动态填报、OCR、草稿、历史、提醒页面的品牌化和触控质感升级。

## 8. CSS And Component Strategy

继续使用现有 Vue 3、Element Plus、 scoped CSS 和 `frontend/src/design/*` token。

不引入新 UI 库，不新增 npm 依赖。优先通过以下文件形成统一语言：

- `frontend/src/design/xt-tokens.css`
- `frontend/src/design/xt-base.css`
- `frontend/src/design/industrial.css`
- 现有 shell、AI、dashboard、entry 页面组件的 scoped CSS

需要避免：

- `transition: all`
- 默认毛玻璃
- 紫蓝 AI 渐变
- 模板式三卡片堆叠
- 纯装饰大背景
- 影响移动端性能的复杂滤镜

## 9. Micro-Interactions

采用组合方案 D。

1. 地图扫线
   - 用在全厂作战地图和管理端总览。
   - 表达 AI 正在巡厂、识别风险。
   - 只动画 transform、opacity 或轻量背景位置。

2. 执行轨推进
   - 用在 AI 自动动作链路。
   - 表达“发现 → 判断 → 执行 → 留痕 → 发布”。
   - 可用当前步骤高亮、完成态勾选、轻微推进动效。

3. 问答转动作
   - 用在 AI 工作台。
   - 用户提问后，回答下方可出现动作卡片。
   - 低风险动作可显示执行状态；高风险动作只进入异常处置或授权边界内的下一步。

4. 思考中动效
   - AI 思考时显示多阶段状态，而不是单一 loading。
   - 状态建议为：读取现场、核对规则、推演影响、生成动作。
   - 动效应克制，不能让回答区抖动或改变布局高度过大。

所有动效需要支持 `prefers-reduced-motion`。

## 10. Data And Behavior Boundaries

前端可以先使用现有 API、store 和 mock 数据完成视觉落地，但必须清楚区分真实数据和 mock 数据。

禁止为了视觉效果伪造业务成功状态。若后端缺少数据，使用真实空状态、待接入状态或现有 mock 管理方式。

AI 自动执行能力必须遵守：

- 后端权限控制。
- 审计日志。
- workflow dispatcher。
- 现有 AI 校验和日报发布边界。

## 11. Implementation Phases

Phase 1: Brand foundation and global shell

- 建立风格化 logo 组件或统一 logo 样式。
- 调整全局 token：色彩、阴影、半径、数字字体、状态色。
- 更新登录页、管理端 shell、移动端顶栏的品牌区。
- 重铺基础卡片、表格、按钮、状态、页面头部的共享视觉语言。

Phase 2: Factory map home and AI manager

- 在管理端总览实现全厂作战地图布局。
- 加入 AI 总管侧舱和执行轨。
- 保持现有 API 和路由不变。

Phase 3: AI workstation polish

- 保留极简问答主路径。
- 丰富思考中、工具调用、动作生成和执行完成动效。
- 增加问答转动作的视觉结构。

Phase 4: Wide page-level rollout

- 大面积重铺主要 `/manage` 页面视觉，包括总览、工厂看板、日报、质量、差异、考勤、导入、主数据、班次详情。
- 收敛 `/entry` 移动端品牌细节和触控质感。
- 清理残余旧 CSS 债务。

每个 phase 可以覆盖较多文件，但必须保持 frontend-only、可构建、可回滚。实现节奏是大范围视觉返工，不是只改一两个局部。

## 12. Testing And Verification

前端相关改动必须至少执行：

- `cd frontend && npm run build`
- 浏览器检查登录页、管理端首页、AI 工作台、移动端入口。

若修改影响 E2E 选择器或路由，需要执行相关 Playwright 测试。

若只改设计文档或 AGENTS/CLAUDE 规则，不要求跑前端构建。

视觉验收点：

- 首屏能看出“鑫泰铝业 MES”，不是通用后台。
- AI Brain 看起来像能管理全场的职业经理人，而不是普通聊天框。
- AI 工作台仍有极简问答路径。
- 思考中状态有质感但不影响阅读。
- `/entry` 仍符合手机优先和一线工人低认知负担。

## 13. Rollback

所有改动按 phase 提交。若某一阶段视觉或构建失败，回滚该阶段提交，不影响之前已通过的 token、shell 或业务代码。

不在同一提交中混合大规模业务逻辑和视觉改造。
