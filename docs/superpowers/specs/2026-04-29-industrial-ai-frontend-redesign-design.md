# 鑫泰铝业 数据中枢工业 AI 前端重构设计

Date: 2026-04-29

## 1. Goal

把前端从普通后台样式升级为“鑫泰铝业 数据中枢”的工业 AI 管理产品：全厂经营控制面、AI 总管侧舱、自动执行轨、岗位直录链路和领导端发布闭环要形成同一套高完成度视觉语言。

本轮升级为 frontend-only 大面积视觉返工，不是局部修补。目标是让登录页、管理端、总览、AI 工作台、主要业务页面和移动端入口形成同一套高完成度工业 AI 产品气质。

本轮设计只定义前端视觉、交互和实施边界，不改变权限、排班、设备绑定、外部 MES/产线系统同步、日报发布、审计日志、readyz、hard gate、AI 校验逻辑。

系统命名规则：

- 正式系统名统一为“鑫泰铝业 数据中枢”。
- UI、规格、计划、文档和对外文案都使用“数据中枢”作为产品身份。
- 不把本产品称为 MES。MES 只作为外部生产系统、数据源、适配器或边界说明出现。

## 2. Visual Thesis

鑫泰铝业 数据中枢的第一屏应像一个真实工业 AI 生产运营中枢：中央看全厂和产线节奏，右侧做预测、判断和派工，底部展示系统正在自动执行的闭环。视觉要有铝业、熔铸、产线、批次流的品牌记忆点，同时保持企业生产系统需要的清晰、克制、精密和高密度可扫描。

质量基准采用仓库全局规则中的前端要求。用户原话“像opus 4.7在Claude code编码设计前端一样，参考其设计前端多好看，你gpt5.5设计的前端太难看了”是本轮的硬约束：必须避免通用 GPT 式模板 UI，追求 Opus 级的构图、品牌、交互细节和浏览器验证完成度。

这意味着每个改动都要回答一个问题：它是否让系统更像真实、昂贵、可用的工业 AI 产品，而不是默认后台皮肤。

本轮升级采用 UI/IA 双层重构：UI 负责现代高科技工业视觉，IA 负责导航、模块、字段和页面信息密度治理。不能只做皮肤；如果字段仍然平铺、导航仍然冗长、模块仍然像通用卡片，视觉升级视为不合格。

## 3. Design Direction

已锁定方向：数据中枢高密度工业控制台，参考 Stripe 式精密产品感。

核心特征：

- 产品身份：鑫泰铝业 数据中枢，不是 MES。
- 密度：指标条、账本式表格、发布链路、预测判断、自动动作并列出现，避免大面积空白。
- 高级感：冷白画布、石墨文字、工业蓝主强调、细线分隔、低半径、克制阴影、数字等宽对齐。
- 工业感：产线拓扑、三维铝锭/设备抽象、班次节奏、岗位直录、产线同步、日报发布、设备绑定和批次流成为界面语言。
- AI 总管：固定承载预测、判断、派工、发布和问答助手入口，像职业经理人而不是普通聊天框。
- 页面延展：其他页面沿用同一套视觉语法，但不机械复制总览首屏。
- 信息压缩：用短导航、模块图形、字段分层和折叠高级区降低冗余，而不是把所有字段继续平铺。

五层治理任务：

1. 科技工业视觉层
   - 强化现代高科技工业感：冷白精密界面、深色 AI 控制区、三维等距产线、微图表、状态流光、AI 脉冲。
   - 继续使用现有 Vue、Element Plus、CSS token，不引入新 UI 库。

2. 导航栏排版层
   - 导航从“文字菜单”升级为“数据中枢命令轨”。
   - 一级只保留：总览、工厂、填报、AI、管理。
   - 二级按当前路由分组展开：总览中心、审阅处置、经营与智能、数据与模板、权限治理、运行保障。
   - 长标题在导航中使用短名，例如“数据接入与字段映射中心”显示为“数据接入”；页面标题保留完整业务名称。

3. 功能模块排版层
   - 每个正式中心使用统一模块入口版式：编号、业务图形、3 个关键指标、当前状态、主动作。
   - 解决所有模块都像通用卡片的问题。

4. 车间图形层
   - 各车间按工艺图形区分：铸锭、热轧、冷轧、拉矫、精整、在线退火、成品库、跨车间链路。
   - 图形进入工厂看板、车间详情、主数据模板、移动端班次摘要。

5. 字段排版与冗余治理层
   - 不删后端字段，不改 API。
   - 前端展示分三层：主字段、辅助字段、审计字段。
   - 主字段一屏必须看到；辅助字段折叠或按业务分组；审计字段放在详情页底部或高级区。
   - 日报详情、质量详情、差异详情、动态填报和班次填报必须优先治理字段平铺问题。

核心画面：

- 中央：全厂/车间/产线控制面，抽象表达熔炉、机台、铸锭、批次流、岗位直录和发布窗口。
- 右侧：AI 总管侧舱，展示预测、判断、已执行动作、下一步方向。
- 底部：自动执行轨，固定表达“发现 → 判断 → 执行 → 留痕 → 发布”。
- 顶部：品牌化鑫泰数据中枢区域，不再只有单字方块 logo。

图形与三维感规则：

- 首屏和工厂看板允许使用轻量等距三维图形：铝锭堆叠、机台块体、产线地台、批次流轨道。
- 图形必须承载业务含义，不能只是装饰背景。每个图形至少对应产线、设备、批次、同步、执行或 AI 推演中的一种语义。
- 三维感优先用 CSS `perspective`、等距变换、分层阴影和几何块实现，避免在管理端大面积引入重 WebGL。
- 若后续真的需要可交互 3D 场景，只在登录页展示面或工厂看板局部使用 Three.js，并按浏览器截图和移动端性能验证通过后保留。
- 普通列表、表格、填报页不强行三维化，只继承低半径、细线、状态、动效和品牌图形语言。

页面级沿用规则：

- 管理总览和工厂看板使用最高密度控制面。
- 报表、质量、差异、考勤、导入、主数据等管理页使用“账本式页面”：紧凑页头、细分区、密集表格、状态胶囊、右侧或底部动作轨。
- 详情页使用“业务档案式页面”：关键结论在前，字段分组紧凑，不堆解释文字。
- AI 工作台使用“对话 + 动作面板”结构：保留极简问答，增强思考中、工具调用和动作生成。
- 移动端 /entry 使用轻量版品牌和大触控控件，不引入复杂地图。

整体 UI 排版规则：

- 页面统一采用“顶部命令区 + 主工作区 + 右侧/底部状态区”的工作台结构。
- 顶部命令区只放页面标题、关键筛选、刷新/主动作；不堆解释文案。
- 主工作区按 12 栅格组织，关键图形和核心表格占主要宽度，辅助列表和 AI 判断放右侧或底部。
- 每页第一屏必须有一个强视觉锚点：工厂图形、模块图形、车间图形、AI 控制区或关键账本表。
- 卡片只用于重复项、模块入口、详情分区或动作面板；禁止卡片套卡片。
- 字体层级不超过 4 级：页面标题、区块标题、字段标签、数值/正文。
- 字段标签短名优先，完整解释放到高级区或文档，不在主界面铺开。

导航排版规则：

- 一级命令轨使用短标签和图形化状态，避免长文本挤压。
- 二级导航可显示完整业务意图，但单项名称不超过 8 个汉字；必要时用短名 + 页面内完整标题。
- 管理端导航分组必须稳定，不随页面内容变化跳动。
- 当前页面状态用图形高亮、细线和状态点表达，不用大面积底色。
- 兼容/历史入口不进入一级导航，只通过重定向或搜索可达。

功能模块图形规则：

- 每个正式中心必须有自己的业务图形，不能全部复用同一种卡片或同一种图标。
- 图形要表达模块职责，而不是做装饰：
  - 系统总览 / 工厂看板：全厂流向、节点、产线控制面。
  - 填报入口 / 填报流程：岗位直录、表单堆叠、批次流。
  - 数据接入：来源环、字段映射、同步线。
  - 审阅中心 / 质量告警：异常分诊、状态列表、处置队列。
  - 日报交付：生成、归档、发布轨道。
  - 成本效益：矩阵、堆叠条、吨耗归集。
  - AI 总管：推演环、思考条、动作卡。
  - 运维 / 治理 / 主数据：矩阵、权限边界、模板堆叠。
- 模块入口图形允许轻量动画，但只能表达状态推进、同步、分析或执行。

功能模块排版规则：

- 模块入口采用统一结构：模块编号、短标题、业务图形、3 个关键指标、当前状态、主动作。
- 模块卡主视觉占卡片高度 35%-45%，指标和动作占 55%-65%，避免只有图形没有信息。
- 模块编号保持两位数，例如 01、03、05、06，不使用“第一部分”类解释文字。
- 模块状态用状态胶囊表达，例如 在线、排队、只读、待接入、已发布、需跟进。
- 主动作只保留一个，其他动作进入更多菜单或页面内操作区。

车间图形规则：

- 各车间看板和车间详情应有差异化工艺图形，避免所有车间都是同一张表。
- 第一批车间/工艺图形：
  - 铸锭 / 铸造：炉区、浇铸、铝锭流。
  - 热轧：热坯、轧辊、剪切。
  - 冷轧 2050 / 1850 / 1450：卷材、辊组、厚度节奏。
  - 拉矫：洗拉、拉矫、退火的连续线。
  - 精整 / 剪切：纵剪、横剪、包装。
  - 在线退火：新厂、园区、连续退火线。
  - 成品库：入库、发货、库存堆叠。
  - 跨车间链路：批次号驱动的事实链。
- 车间图形可出现在工厂看板、车间看板、车间详情、主数据模板和移动端当前班次摘要中。
- 移动端使用简化图形，不展示复杂三维地台。

字段排版与字段冗余规则：

- 前端展示字段分为三层：
  - 主字段：业务判断必须立即看到，如产量、投入、成材率、班次、状态、发布时间、是否可交付。
  - 辅助字段：帮助解释主字段，如生成范围、输出方式、质量闸门说明、异常摘要、字段来源。
  - 审计字段：追溯字段，如编号、生成时间、归档确认人、归档确认时间、API 来源、fallback 标识。
- 主字段使用 KPI、账本行、状态胶囊或短字段组，不使用大面积 `el-descriptions` 平铺。
- 辅助字段按业务分组折叠：生产、质量、交付、异常、来源。
- 审计字段默认放在“审计与来源”折叠区，详情页底部可展开。
- 移动端每屏最多放 4 个主输入字段；补录字段、线索字段、审计字段进入折叠区或滑屏页。
- 字段名称要短，避免解释字段名。示例：用“发布”替代“发布时间状态说明”，用“来源”替代“归档确认来源说明”。
- 字段不能为了视觉被删除；只做分组、折叠、排序、短名和视觉层级调整。

## 4. AI Brain Product Role

AI Brain 不是普通聊天助手，而是“总管全场的职业经理人”。

它在界面中的职责：

- 预测：提前发现产线节奏、填报缺口、日报窗口和同步偏差。
- 分析：解释异常原因和影响范围。
- 给方向：给出下一步处理方向。
- 真正做事：在规则允许范围内自动催报、重新校验、生成摘要、触发发布或记录动作。
- 留痕：所有自动动作必须能被审计和追溯。

本设计不新增越权执行能力。前端只呈现已有或后续明确授权的动作，不绕过后端权限、审计或 workflow dispatcher。

## 5. AI Workstation Constraint

必须保留现有 AI 工作台的极简问答助手方式。

AI 工作台不是全部改造成复杂地图页。它应继续保持：

- 输入框清楚，问答路径短。
- 对话消息易读，像高质量生产工具，不像营销展示页。
- 支持 AI 辅助操作面板，把回答进一步转成可执行动作。

允许增强：

- 思考中状态增加更丰富的动效，例如呼吸式状态点、分段推理轨、工具调用闪烁、答案生成进度。
- 工具调用状态明确区分“分析中、查数据、生成动作、执行中、已完成”。
- 问答转动作时，动作卡片从回答下方自然展开，不打断主对话流。

不允许：

- 删除极简问答主路径。
- 把 AI 工作台改成只有地图或大屏。
- 用大段说明文字解释功能。
- 添加与业务无关的装饰动画。

## 6. Brand And Logo System

品牌系统需要从“鑫”字方块升级为一套可复用的图形语言。

图形元素：

- 铝锭：几何块体、倒角、冷白金属边。
- 熔炉：内核暖铜色，只作为小面积能量点。
- 批次流：细线、节点、箭头、轨道。
- AI 中枢：克制的环形扫描或聚合节点。

使用原则：

- Logo 图形可在登录页、管理端侧栏、移动端顶栏、AI Brain 页使用。
- 不使用复杂插画，不使用大面积渐变背景。
- 品牌色以冷白、石墨、工业蓝为主，小面积铜色代表熔铸能量。
- 图形必须在 28px、32px、44px 和登录页大尺寸下都可读。

## 7. Surface Plan

本轮是大面积视觉返工，但仍只在前端范围内推进。

第一批重构页面：

1. 登录页
   - 强化品牌第一印象。
   - 使用风格化 logo 和工厂作战地图的静态缩影。
   - 保持登录表单清楚，不做营销落地页。

2. 管理端 shell
   - 侧栏、顶栏、命令入口和品牌区统一。
   - 从普通导航升级为数据中枢工业控制台框架。
   - 保持现有路由、权限和导航来源。

3. 管理端总览
   - 使用全厂作战地图作为第一屏核心。
   - 右侧加入 AI 总管侧舱。
   - 底部加入自动执行轨。

4. AI Brain / AI 工作台
   - 保留极简问答助手。
   - 增强思考中、工具调用、执行动作、完成状态。
   - 问答与动作面板并存，优先保证对话可读。

5. 移动端 /entry
   - 使用同一品牌系统的轻量版本。
   - 不引入复杂地图。
   - 继续保持大按钮、大字、一线工人低认知负担。

第二阶段再扩展：

- 各 `/manage` 模块页的卡片、表格、筛选器、空状态、批量操作条全部沿用数据中枢视觉语言。
- Dashboard、导入、主数据、班次详情、质量、日报、差异、考勤等页面的视觉重铺。
- 移动端 `/entry`、动态填报、OCR、草稿、历史、提醒页面的品牌化和触控质感升级。

## 8. CSS And Component Strategy

继续使用现有 Vue 3、Element Plus、 scoped CSS 和 `frontend/src/design/*` token。

不引入新 UI 库，不新增 npm 依赖。优先通过以下文件形成统一语言：

- `frontend/src/design/xt-tokens.css`
- `frontend/src/design/xt-base.css`
- `frontend/src/design/industrial.css`
- 现有 shell、AI、dashboard、entry 页面组件的 scoped CSS

需要避免：

- `transition: all`
- 默认毛玻璃
- 紫蓝 AI 渐变
- 模板式三卡片堆叠
- 纯装饰大背景
- 影响移动端性能的复杂滤镜
- 大面积空白导致的“演示稿感”
- 把产品身份写成外部生产系统

## 9. Micro-Interactions

采用组合方案 D。

1. 地图扫线
   - 用在全厂作战地图和管理端总览。
   - 表达 AI 正在巡厂、识别风险。
   - 只动画 transform、opacity 或轻量背景位置。

2. 执行轨推进
   - 用在 AI 自动动作链路。
   - 表达“发现 → 判断 → 执行 → 留痕 → 发布”。
   - 可用当前步骤高亮、完成态勾选、轻微推进动效。

3. 问答转动作
   - 用在 AI 工作台。
   - 用户提问后，回答下方可出现动作卡片。
   - 低风险动作可显示执行状态；高风险动作只进入异常处置或授权边界内的下一步。

4. 思考中动效
   - AI 思考时显示多阶段状态，而不是单一 loading。
   - 状态建议为：读取现场、核对规则、推演影响、生成动作。
   - 动效应克制，不能让回答区抖动或改变布局高度过大。

所有动效需要支持 `prefers-reduced-motion`。

展示级动效是本设计语言的一部分，需沿用视觉稿中已经确认的感觉：

- 产线流光：在工厂控制面、执行轨和关键链路上使用水平流动高光，表达数据和批次正在向前推进。
- AI 脉冲：AI 总管入口使用克制的聚合点/扫描脉冲，表达系统正在推演，而不是普通装饰图标。
- 三段思考条：问答助手思考中使用多段条形动效，表达“读数据 → 核规则 → 生成动作”的过程。
- 状态卡入场：预测、判断、动作、复盘卡片可以有轻微 `opacity + translateY` 入场，形成职业经理人持续输出的感觉。
- 执行推进：催办、校验、汇总、发布等动作以小幅推进、完成态亮起和状态胶囊变化表达闭环。
- 三维悬浮：等距产线、铝锭和 AI 中枢可以有非常轻的 z-depth 浮动或脉冲，但不能旋转炫技。

动效约束：

- 动画只服务状态、层级和执行感，不做无意义漂浮装饰。
- 优先动画 `transform`、`opacity`、轻量背景位置，不使用 `transition: all`。
- 动画时长以 120ms、180ms、260ms 为主，循环动效要慢、稳、低对比。
- 页面整体不能因为动效造成布局跳动、文本抖动或移动端性能下降。

## 10. Data And Behavior Boundaries

前端可以先使用现有 API、store 和 mock 数据完成视觉落地，但必须清楚区分真实数据和 mock 数据。

禁止为了视觉效果伪造业务成功状态。若后端缺少数据，使用真实空状态、待接入状态或现有 mock 管理方式。

AI 自动执行能力必须遵守：

- 后端权限控制。
- 审计日志。
- workflow dispatcher。
- 现有 AI 校验和日报发布边界。

## 11. Implementation Phases

Phase 1: Brand foundation and global shell

- 建立风格化 logo 组件或统一 logo 样式。
- 调整全局 token：色彩、阴影、半径、数字字体、状态色。
- 更新登录页、管理端 shell、移动端顶栏的品牌区。
- 重铺基础卡片、表格、按钮、状态、页面头部的共享视觉语言。
- 将系统身份统一为“鑫泰铝业 数据中枢”。
- 建立导航短名、模块入口、字段分层和页面排版规则。

Phase 2: Factory map home and AI manager

- 在管理端总览实现全厂作战地图布局。
- 加入 AI 总管侧舱和执行轨。
- 保持现有 API 和路由不变。

Phase 3: AI workstation polish

- 保留极简问答主路径。
- 丰富思考中、工具调用、动作生成和执行完成动效。
- 增加问答转动作的视觉结构。

Phase 4: Wide page-level rollout

- 大面积重铺主要 `/manage` 页面视觉，包括总览、工厂看板、日报、质量、差异、考勤、导入、主数据、班次详情，并沿用同一套数据中枢工业密度语言。
- 对日报详情、质量详情、差异详情、动态填报、班次填报执行字段分层治理，减少字段平铺和解释文案。
- 收敛 `/entry` 移动端品牌细节和触控质感。
- 清理残余旧 CSS 债务。

每个 phase 可以覆盖较多文件，但必须保持 frontend-only、可构建、可回滚。实现节奏是大范围视觉返工，不是只改一两个局部。

## 12. Testing And Verification

前端相关改动必须至少执行：

- `cd frontend && npm run build`
- 浏览器检查登录页、管理端首页、AI 工作台、移动端入口。

若修改影响 E2E 选择器或路由，需要执行相关 Playwright 测试。

若只改设计文档或 AGENTS/CLAUDE 规则，不要求跑前端构建。

视觉验收点：

- 首屏能看出“鑫泰铝业 数据中枢”，不是通用后台，也不是 MES 产品。
- 其他页面能沿用同一套数据中枢视觉语言：低半径、细线、密集账本、工业蓝、产线/执行链路语义。
- 首屏和工厂看板有足够图形、UI 和动效记忆点，包括等距三维产线、铝锭/设备块体、AI 中枢脉冲和执行流光。
- 导航栏短、稳、可扫；一级命令轨不出现长标题挤压。
- 功能模块入口包含编号、图形、3 个指标、状态和主动作。
- 详情页和填报页没有大面积字段平铺；主字段、辅助字段、审计字段有清晰层级。
- AI Brain 看起来像能管理全场的职业经理人，而不是普通聊天框。
- AI 工作台仍有极简问答路径。
- 思考中状态有质感但不影响阅读。
- `/entry` 仍符合手机优先和一线工人低认知负担。

## 13. Rollback

所有改动按 phase 提交。若某一阶段视觉或构建失败，回滚该阶段提交，不影响之前已通过的 token、shell 或业务代码。

不在同一提交中混合大规模业务逻辑和视觉改造。
