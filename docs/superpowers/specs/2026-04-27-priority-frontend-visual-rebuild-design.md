# 2026-04-27 前端优先页面视觉重构设计

## 1. 背景与目标

本轮负责 aluminum-bypass 前端优先页面视觉重构，并同步收口录入端主线索与登录入口语义。

本轮优先页面：

- `/login`
- `/entry`
- `/entry/report/*`
- `/entry/advanced/*`
- `/review/brain`

目标是在**不改后端接口、不改数据库、不改 payload 结构、不改后端权限模型**的前提下，把上述页面尽量还原到 `docs/ui-reference/highres/` 高清目标图，并把视觉差异尽量压到 5% 以内；同时减少一线输入负担，为微信内打开和后续微信小程序化预留更轻量的交互路径。

## 2. 范围、约束与字段边界

### 2.1 本轮范围与不可变更项

本轮聚焦前端视觉重构与前端交互收口，允许调整前端页面语义和登录流呈现，但**不修改后端接口、数据库、payload 结构和后端权限模型**。

本轮允许：

- 前端页面结构重排
- 前端展示语义调整
- 登录后按账号权限自动跳转前端落点
- 将“批次号 / 随行卡号”在前端界面上收口为单一主入口语义
- 为微信内打开尺寸与后续小程序化预留更轻量交互

本轮不允许：

- 修改后端接口 contract
- 修改数据库字段
- 修改提交 payload 结构
- 修改后端权限模型
- 正式完成 MES 对接
- 在本轮直接完成“批次号 / 随行卡号”底层 contract 合并

### 2.2 字段保护

必须保留并保持原义：

- 车间
- 产线/机台
- 班次
- 业务日期
- 合金牌号
- 批次号
- 要求规格
- 上机时间
- 下机时间
- 上机重量
- 下机重量
- 废料
- 成品率
- 异常说明
- 班末补充确认
- `operator_notes`

### 2.3 MES 边界

MES 未对接阶段采用“**批次号驱动的半自动填报**”模式，避免纯手填，也避免伪装成已经打通 MES。

#### 本轮（Phase 1）原则

- 前端界面只突出“批次号”这一个主入口
- 当前业务上将“批次号”视为现场唯一主线索
- 若现有底层仍存在“随行卡号”字段，本轮仅做前端语义收口，不改后端 contract
- 一线若扫描 MES 二维码，可先解析并填入已有线索字段，但界面主入口仍表现为批次号
- 所有自动带出的字段都必须保留人工修改权限
- 未对接 MES 前，不把任何 MES 线索字段改成必填
- 不伪造自动追踪完成或 MES 已联通状态

#### 本轮（Phase 1）可由本系统半自动回填的字段

在 MES 未对接前，只要本系统能够基于批次号命中上一工序记录，就允许先回填可推断字段，并标注来源为“系统回填 / 待确认”：

- 上机重量（优先取上工序下机重量）
- 合金
- 规格
- 其他当前系统内可稳定推断的字段

这些字段的共同规则：

- 先带出
- 可人工修改
- 提交时仍走现有 payload
- 不改变字段语义

#### 下一轮（Phase 2）正式 contract 合并与 MES 对接目标

下一轮若进入字段 contract 合并与 MES 正式对接，再统一处理：

- “批次号 / 随行卡号”底层 contract 合并
- 扫码结果与存储字段统一
- payload 与后端 schema 收口为单字段语义
- MES 来源字段的正式接入与优先级规则

MES 真正对接后，可由 MES 提供或优先回填的字段包括：

- 产品类型
- 批次号
- 坯料重量
- 合金
- 状态
- 成品规格
- 生产安排的工艺路线

前端当前阶段可以为这些字段预留来源展示位，但文案必须保持“待 MES 对接”或“暂未获取”，不能误导为已自动联通。

#### MES 线索字段现阶段约束

以下字段在当前阶段只能保持“待对接”语义，不允许变成必填，不允许伪造自动追踪：

- 坯料号
- MES 后续码
- 后续对象号
- 来源坯料线索
- 合同号
- 当前所在工艺
- 状态
- 更新时间

说明：

- 本轮前端界面不再把“随行卡号”作为独立主入口展示；若底层仍存在该字段，仅作为兼容映射保留，待 Phase 2 再决定是否彻底移除或合并。
- 主操填写或确认的“规格”属于当前生产录入字段，必须保留并可编辑，不应因为未来 MES 可提供成品规格就被降级成纯待对接字段。
- 未来 MES 可提供的是“成品规格 / MES 来源规格”，前端应通过来源标签或补充展示区区分“当前录入规格”和“MES 来源规格”，避免混淆字段语义。
- 主操不需要看到客户名，也不需要填写客户字段，因此本轮与下一轮都不把“客户”纳入主操录入视图。

#### 废料与成品率规则

废料与成品率采用“**自动计算 + 可人工覆盖**”模式：

- 页面实时计算建议值
- 默认展示当前计算结果
- 主操允许人工修改
- 若人工覆盖，界面应明确标注为“人工修正”或等价语义
- 在 MES 未对接前，主操仍可按现场需要填写这两个字段

### 2.4 `/review/brain` 安全边界

`/review/brain` 仅作为辅助建议入口：

- 允许：问答、建议、证据查看、上下文摘要、导航
- 禁止：AI 自动执行、自动排产、自动关闭告警
- 无真实接口的动作必须 disabled

## 3. 视觉目标

全页统一采用以下视觉语言：

- 浅灰蓝背景
- 白底细边框卡片
- 轻阴影
- 大号蓝色编号
- 中文标题强化
- KPI 数字大
- 表格紧凑
- 导航明显可见
- 间距统一
- 整体脱离“普通后台模板感”

## 4. 改造范围与总体策略

采用“**页面级重构 + 逻辑复用**”策略：

- 保留现有 store、router、API 调用、表单字段绑定、提交逻辑
- 重做页面骨架、视觉层级、模块顺序、响应式规则
- 必要时补少量轻量视觉壳层/样式 token
- 不做大规模抽象，不扩大到非优先页面的全面翻修

改造优先级采用用户确认的“**像图优先**”原则：

- 允许较大幅度重排前端结构
- 以贴近目标图为最高优先级
- 但所有数据 contract 和业务边界保持不变

## 5. 页面级设计

### 5.1 `/login`

目标：从传统后台登录页转成更轻量的权限入口页，并在登录后按账号权限自动跳转对应页面。

设计：

- 顶部展示 `02` 大号编号与中文标题
- 中部主卡展示系统名与精简登录入口
- 不再要求用户手动选择录入端 / 审阅端 / 管理端
- 登录成功后按账号权限自动跳转到对应默认页面
- 页面视觉仍尽量贴近 `02-login.png` 的浅色卡片结构，但去掉三角色选择作为主交互
- 为后续企业微信绑定登录 / 微信小程序自动登录预留更少步骤的入口语义
- 保留现有认证方式、权限判断和默认跳转机制的可兼容实现

验收重点：

- 视觉上明显接近目标图
- 页面主焦点是轻量登录入口，不是传统后台表单
- 登录后按权限自动跳转，不要求角色手选

### 5.2 `/entry`

目标：对齐 `03-entry-home.png` 与 `15-entry-responsive.png`，形成独立填报端首页。

设计：

- 顶部展示 `03` 编号、中文标题、当前班次与身份摘要
- 第一屏突出 3-4 个核心指标：待填、已提交、异常补卡、当前班次
- 主区拆分为：最近提交状态、快捷操作、MES 线索追踪说明
- 批次号作为唯一主线索入口
- 支持扫码带入线索字段，也支持手动填写批次号启动填报
- 明确隐藏 Review/Admin 相关导航
- 强化主操作按钮与移动端可点击面积

验收重点：

- 录入端更像独立终端，不像通用 dashboard
- 主操能快速找到填报入口
- 移动端不出现 Review/Admin

### 5.3 `/entry/report/*`

目标：对齐 `04-entry-flow.png`，从长表单改成轻步骤流程页。

设计：

- 顶部展示 `04` 编号与简化步骤条
- 主表单采用单列优先布局
- 字段按业务分组重排
- 底部固定主操作区：上一步 / 保存草稿 / 下一步或提交
- 桌面端保留辅助说明区；移动端将说明区下沉到表单后
- 强化主按钮，弱化次要说明文字
- 批次号输入后优先尝试按本系统记录回填上一工序可推断字段
- 自动带出字段必须保留人工修改权限，并显示来源标签
- 废料与成品率实时自动计算，但允许人工覆盖

字段分组原则：

- 基础识别（仅突出批次号主线索）
- 重量时间
- 异常补充
- MES 线索说明

验收重点：

- 字段不变但认知负担更低
- 步骤感明显增强
- 不再呈现为普通后台长表单

### 5.4 `/entry/advanced/*`

目标：与 `/entry/report/*` 共用视觉语言，但承载更完整的字段呈现。

设计：

- 保持步骤流布局
- 必保字段按“基础识别 / 重量时间 / 异常补充确认 / MES 线索说明”组织
- `operator_notes` 保持可读可填
- 批次号为唯一主线索，支持扫码预填或手动录入
- 批次号命中本系统记录后，允许回填上一工序可推断字段，但必须可人工修改
- MES 字段明确标注“待对接”，不设必填，不伪造自动追踪
- 废料与成品率实时自动计算，但允许人工覆盖并标记修正来源
- 移动端表单单列优先，底部操作区固定且不遮挡输入

验收重点：

- 高级表单完整但不显得臃肿
- 线索追踪说明在小屏仍可读
- 录入链路可用性优先于堆砌信息

### 5.5 `/review/brain`

目标：把当前中心页简化为极简 Agent 问答入口，对齐 `11-ai-control.png` 的核心语义而非大屏装饰。

设计：

- 页面核心为一个主问答框
- 只保留最多 6 个建议动作
- 独立展示：证据来源、最近上下文摘要、辅助建议 / 系统提示
- 无真实接口的动作统一 disabled
- 去掉重驾驶舱化的多块 KPI 与伪自动控制模块
- 明确所有动作都是辅助建议或导航，不代表自动执行

验收重点：

- 问答框成为绝对主视觉
- 建议动作不超过 6 个
- 不出现自动执行、自动排产、自动关闭告警语义

## 6. 组件与实现策略

### 6.1 组件策略

优先复用现有逻辑，在前端层补轻量参考图风格壳：

- 页面壳：统一编号标题、顶部工具区、主内容区
- 卡片壳：白底、细边框、轻阴影、统一内边距
- KPI 卡片：大数字、紧凑趋势、弱说明
- 移动端底部操作区：确保主按钮可见且不遮挡内容
- Brain 问答区：主输入框 + 建议动作 + 证据/上下文区

若现有通用组件足够，则优先套壳；若不足，再补少量薄组件或页面内局部组件。

### 6.2 数据流策略

严格复用现有数据 contract：

- 不改 API
- 不改字段名
- 不改 payload
- 不改 store contract
- 不改权限判断语义

允许的变更仅限：

- 重新组织前端 view model
- 调整模块展示顺序
- 增加只读状态标签、来源标签与禁用态表达
- 在移动端把宽表格转成卡片/纵向块展示
- 前端界面把“批次号 / 随行卡号”收口为单一主入口语义，但保持旧 contract 兼容

## 7. 响应式设计

重点验收尺寸：

- 375（微信内常见窄屏）
- 390
- 414
- 768

规则：

- `/entry`、`/entry/report/*`、`/entry/advanced/*` 默认单列优先
- 布局优先适配微信内打开的有效可视区域
- 所有 grid/flex 子项显式允许收缩，避免横向撑破
- 宽内容在小屏改为卡片流或纵向字段块
- 底部操作区预留安全间距，避免遮挡输入
- 小屏保持线索追踪说明可读
- `fill-only` 用户在移动端不显示 Review/Admin
- 交互应为后续微信小程序化保留低输入负担与少步骤特征

## 8. 测试与验证策略

采用先测试后实现的顺序，锁定结构和边界：

1. 为 `/login`、`/entry`、`/entry/report/*`、`/entry/advanced/*`、`/review/brain` 增补或更新 E2E / smoke 断言
2. 覆盖移动端 `375 / 390 / 414 / 768`
3. 对字段保护做断言：
   - 指定字段仍存在
   - 字段含义与提交流程不变
   - 批次号可作为唯一主线索启动填报
   - 批次号命中记录后可回填上一工序可推断字段
   - 自动带出字段仍允许人工修改
   - MES 字段仍为“待对接”语义
   - 不新增 MES 必填
   - 废料、成品率自动计算但允许人工覆盖
4. 对 `/login` 做边界断言：
   - 登录后按权限自动跳转
   - 不要求用户手动选择角色入口
5. 对 `/review/brain` 做边界断言：
   - 主问答框存在
   - 建议动作不超过 6 个
   - 无真实接口动作 disabled
   - 不出现自动执行/自动排产/自动关闭告警语义

## 9. 完成标准

只有在以下条件满足后，才能声明本轮实现完成：

- 目标页面已明显更接近高清目标图
- 明确确认：本轮没有改 payload / 后端；字段 contract 合并留到下一轮
- `/review/brain` 已变成简约 Agent 问答入口
- 登录后已按权限自动跳转
- 移动端 `375 / 390 / 414 / 768` 不横向撑破
- `frontend build` 通过
- 相关 `e2e` 通过
- 若本轮覆盖到 Python 主链路验证，则如实运行并汇报 `pytest` 结果
- 最终输出仍未达到 5% 的页面与剩余风险

## 10. 非目标

以下内容不在本轮范围内：

- 修改后端模型、数据库表或 migration
- 正式接通 MES 后续追踪链路
- 在本轮直接完成“批次号 / 随行卡号”底层 contract 合并
- 新增 AI 自动执行能力
- 把非优先页面全部一起重构
- 用大规模组件体系翻修替代当前页面优先交付

## 11. 已确认决策

- 优先级选择：像图优先
- `/entry/report/*` 与 `/entry/advanced/*` 允许重组为步骤感更强的单列流程布局，但字段名、字段含义、提交 payload 完全不变
- 本轮（Phase 1）前端界面将“批次号 / 随行卡号”收口为单一主入口语义，只突出批次号；底层 contract 合并留到下一轮
- MES 未对接阶段采用“批次号驱动的半自动填报”模式：批次号为唯一主线索，系统可基于批次号回填上一工序可推断字段，且所有回填字段保留人工修改权
- 废料、成品率采用“自动计算 + 可人工覆盖”方案
- `/review/brain` 采用极简问答入口方案
- `/login` 改为登录后按权限自动跳转，不再要求用户手动选择角色入口
- 移动端优先适配微信内打开，并为后续微信小程序化与微信绑定自动登录预留更轻量的交互方向
