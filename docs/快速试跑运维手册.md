# 快速试跑运维手册

这份手册只服务当前 `Phase 1 一个车间快速试跑`。

目标不是做复杂运维，而是让常见动作都能按固定命令完成。

## 1. 首次部署

在云主机项目目录执行：

```bash
./scripts/deploy_trial.sh
./scripts/deploy_trial.sh --dry-run   # 只做环境预检，不执行发布与健康检查
```

它会做：

1. 检查基础文件与运行环境：`docker-compose.yml` / `docker-compose.prod.yml` 存在，且 `docker` 命令存在、`docker compose` 可用并可连接到 docker daemon，同时安装了 `curl`（健康检查依赖）。
2. 检查 `.env` 是否存在，并校验关键配置：`APP_ENV` 必须为 `production`，`POSTGRES_PASSWORD` / `SECRET_KEY` / `INIT_ADMIN_PASSWORD` / `CORS_ORIGINS` 必填，且不能包含 `CHANGE_ME` 占位符；同时会拦截弱值（`SECRET_KEY` 长度需 >= 32，`INIT_ADMIN_PASSWORD` 长度需 >= 12，且两者都不能是默认/示例值）。
3. 检查 compose 配置前，先校验 SSL 证书文件 `ssl/cert.pem` 与 `ssl/key.pem` 是否存在且非空，并校验内容格式：`ssl/cert.pem` 需包含 `BEGIN CERTIFICATE`，`ssl/key.pem` 需包含 `BEGIN` 与 `PRIVATE KEY`（兼容 RSA/EC/PKCS8）。
4. 检查 compose 配置
5. 启动 `db / backend / nginx`
6. 输出当前容器状态
7. 自动执行一次 `check_trial_stack.sh`，并确认 `hard_gate_passed=true`；若失败会自动输出关键容器状态与后端/nginx日志片段

如需指定检查地址，可设置：

```bash
TRIAL_BASE_URL=https://你的域名 ./scripts/deploy_trial.sh
```

## 2. 健康检查

部署后执行：

```bash
./scripts/check_trial_stack.sh https://你的域名
```

也可以不传参数，默认检查 `https://localhost`：

```bash
./scripts/check_trial_stack.sh
./scripts/check_trial_stack.sh --dry-run  # 仅校验参数与依赖，不发起网络探测
```

它会检查：

1. 检查基础文件与运行环境：`docker-compose.yml` / `docker-compose.prod.yml` 存在、`.env` 已准备，且 `docker` 命令存在、`docker compose` 可用、`docker daemon` 可连接、`curl` 已安装
2. `/healthz` 与 `/readyz` 的就绪状态（默认最多重试 `24` 次、每次间隔 `5` 秒，可通过 `READY_RETRIES` / `READY_INTERVAL_SECONDS` 覆盖）
3. `/readyz`，并确认 `hard_gate_passed=true`；若失败会自动输出关键容器状态与后端/nginx日志片段
4. `/` 首页可访问（单次请求超时 `15` 秒），并校验返回内容包含前端挂载锚点 `id="app"`
5. `/entry` 可访问（单次请求超时 `15` 秒），并校验返回内容包含前端挂载锚点 `id="app"`
6. `/manage/factory` 可访问（单次请求超时 `15` 秒），并校验返回内容包含前端挂载锚点 `id="app"`
7. `/api/v1/auth/login` 入口可达（只检查路径响应不为 404、000 或 5xx，不校验登录结果；若返回 5xx 说明服务异常）
8. 当前 compose 服务状态
9. `db / backend / nginx` 是否都在运行

## 2.1 一键上线闸门（推荐）

当需要给出“可上线 / 不可上线”的明确结论时，直接执行：

```bash
./scripts/go_live_gate.sh https://你的域名
```

默认会依次检查：

1. 站点就绪（`check_trial_stack.sh`）
2. 试点配置硬门槛（`check_pilot_config.py --json`）
3. AI live 探针（助手问答/出图 + 领导摘要）
4. 角色权限 smoke（reviewer/mobile scope/write guard）
5. 回滚预检（`backup_db.sh --dry-run` + `restore_db.sh --dry-run`）

可选参数：

```bash
./scripts/go_live_gate.sh --dry-run
./scripts/go_live_gate.sh --date 2026-04-22
./scripts/go_live_gate.sh --skip-ai
./scripts/go_live_gate.sh --skip-role-smoke
```

## 2.2 小白友好的云端一键流程

如果你希望把“更新+发布+闸门”放在一条命令里执行，可用：

```bash
./scripts/launch_cloud_trial.sh https://你的域名 --pull
```

同义执行序列：

1. `deploy_trial.sh`
2. `go_live_gate.sh`（含 stack/pilot/AI/角色 smoke/回滚预检）

常用变体：

```bash
./scripts/launch_cloud_trial.sh --dry-run
./scripts/launch_cloud_trial.sh --pull --skip-ai
./scripts/launch_cloud_trial.sh --skip-pull --date 2026-04-22
```

## 5. 备份数据库

```bash
./scripts/backup_db.sh
./scripts/backup_db.sh --dry-run  # 仅做备份前置预检，不导出数据库
```

备份文件默认会放到 `backups/`。
如传入自定义备份路径，脚本会自动创建对应目录（目录不可写时会直接报错）。
执行前会先检查：compose 文件存在、`.env` 已准备、docker/compose/daemon 可用。
如果备份产物为空，脚本会直接报错并删除空文件，避免把坏备份带入恢复流程。
备份完成后会立即做一次备份格式可读性校验（`pg_restore -l`），损坏备份会当场拦截。
脚本异常中断时会自动清理容器内临时备份文件，避免残留垃圾文件。

## 6. 恢复校验

建议先确认服务运行，再执行恢复：

```bash
./scripts/check_trial_stack.sh
```

再执行恢复：

```bash
./scripts/restore_db.sh backups/你的备份文件.dump
./scripts/restore_db.sh --dry-run backups/你的备份文件.dump  # 仅做恢复前置预检，不执行恢复
```

这个动作会把备份恢复到一个校验库，确认备份不是坏的。
如果 compose 文件缺失、`.env` 缺失、docker 环境未就绪、备份文件不存在/为空或 `db` 未运行，脚本会直接给出可读提示并停止执行。
恢复前还会先做一次备份格式可读性校验（`pg_restore -l`）；若备份损坏会先报错退出，不进入后续恢复。
脚本异常中断时会自动清理容器内临时恢复文件，避免影响后续恢复。
恢复目标库必须是独立校验库，不能与生产库同名，也不能使用 `postgres/template0/template1` 保留库名。
兼容旧口径说明：如果备份文件不存在或 `db` 未运行，也会直接停止并提示。

## 7. 回滚思路

如果新版本试跑异常：

1. 切回上一个 Git 提交
2. 重新执行部署脚本
3. 若怀疑数据库问题，优先先做恢复校验，不直接覆盖生产库

## 8. 本轮不要做的事

1. 不要直接全厂放量
2. 不要在试跑时同时接企业微信正式入口
3. 不要把 `.env`、证书和密钥提交到 GitHub

## 9. AI 助手 API（火山引擎 DeepSeek-V3.2）

在 `.env` 中确认以下配置：

```dotenv
LLM_ENABLED=true
LLM_API_BASE=https://ark.cn-beijing.volces.com/api/v3
LLM_API_KEY=你的火山引擎 API Key
LLM_ENDPOINT_ID=你的火山引擎推理接入点ID
LLM_MODEL=DeepSeek-V3.2
```

注意：火山引擎 Ark 真实请求的 `model` 参数通常应使用接入点 ID；本项目会优先使用 `LLM_ENDPOINT_ID`。若仅配置 `LLM_MODEL=DeepSeek-V3.2`，后端会在首次 `NotFound` 时自动查询 `/models` 并重试匹配 `deepseek-v3-2-*`，但生产建议仍显式填写 `LLM_ENDPOINT_ID`。

启动后可用以下命令做 API 验证：

```bash
python -m pytest backend/tests/test_assistant_routes.py -q
python -m pytest backend/tests/test_aggregator_agent.py backend/tests/test_reconciler_agent.py backend/tests/test_reminder_agent.py backend/tests/test_reporter_agent.py backend/tests/test_validator_agent.py -q
```
