# 部署文档

## 1. 适用范围

本文档用于当前 `aluminum-bypass` 项目的内部试点上云部署。

当前默认部署目标：

1. 一台 Linux 云主机
2. Docker Engine + Docker Compose
3. 公网域名 + HTTPS
4. PostgreSQL 使用 `docker-compose.yml` 内置服务
5. Phase 1 仍按“主操直录 + 专项 owner 补录 + 系统自动汇总 + 管理端审阅”口径上线

当前推荐上线方式：

1. 先做 `一个车间快速试跑`
2. 先开放浏览器访问
3. 企业微信正式入口先不接，优先浏览器 / 钉钉链路
4. 先接 GitHub，再让云主机通过 `git clone / git pull` 更新代码

## 2. 当前上线前提

上线前必须确认以下前提：

1. 后端测试基线通过：`docker compose run --rm backend sh -lc "pytest -q"`
2. 前端构建通过：`cd frontend && npm run build`
3. `/readyz` 返回 `hard_gate_passed=true`
4. 现场主数据已完成：车间、班次、角色、模板、专项 owner、管理员账号
5. 生产 `.env` 已准备完成，且未提交到 Git
6. 域名、证书、正式身份入口配置已确认；若本轮只走浏览器 / 钉钉，企业微信可信地址和回调地址不作为当前阻塞

## 3. 必备文件

当前生产部署依赖以下文件：

1. `docker-compose.yml`
2. `docker-compose.prod.yml`
3. `.env` 或等效生产环境变量文件
4. `ssl/` 目录中的正式证书文件

## 4. 必填环境变量

至少需要填写：

```env
APP_ENV=production
POSTGRES_USER=
POSTGRES_PASSWORD=
POSTGRES_DB=
SECRET_KEY=
INIT_ADMIN_USERNAME=
INIT_ADMIN_PASSWORD=
INIT_ADMIN_NAME=系统管理员
DEFAULT_TIMEZONE=Asia/Shanghai
VITE_API_BASE_URL=/api/v1
```

按当前 Phase 1 推荐值：

```env
MES_ADAPTER=null
WORKFLOW_ENABLED=false
AUTO_PUBLISH_ENABLED=true
AUTO_PUSH_ENABLED=true
AUTO_PIPELINE_REQUIRE_READY=true
WECOM_APP_ENABLED=false
WECOM_BOT_ENABLED=false
LLM_ENABLED=false
APP_CONNECTION_ENABLED=false
```

说明：

1. Phase 1 不以 MES 联调为上线前提，所以 `MES_ADAPTER` 可继续保持 `null`
2. 若企业微信暂未正式联通，可先保持 `WECOM_APP_ENABLED=false`，先跑浏览器 / 内网试点
3. 若现场需要先静默试跑，可临时设：
   - `AUTO_PUSH_ENABLED=false`
   - `AUTO_PUBLISH_ENABLED=true`

## 5. 云主机准备

建议最低配置：

1. 2 vCPU
2. 4 GB RAM
3. 40 GB SSD
4. Ubuntu 22.04 LTS

主机准备步骤：

```bash
sudo apt update
sudo apt install -y docker.io docker-compose-plugin git
sudo systemctl enable docker
sudo systemctl start docker
sudo usermod -aG docker $USER
```

重新登录后验证：

```bash
docker --version
docker compose version
```

## 6. 代码部署

将仓库放到目标目录，例如：

```bash
mkdir -p /srv/aluminum-bypass
cd /srv/aluminum-bypass
git clone <你的仓库地址> .
```

后续试跑版更新建议统一走：

```bash
cd /srv/aluminum-bypass
git pull
```

准备生产配置：

```bash
cp .env.example .env
mkdir -p ssl
```

将正式证书放入 `ssl/`，并编辑 `.env`。

## 7. 启动命令

先做 compose 配置检查：

```bash
docker compose -f docker-compose.yml -f docker-compose.prod.yml config
```

生产启动：

```bash
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d --build
```

查看状态：

```bash
docker compose ps
docker compose logs backend --tail 200
docker compose logs nginx --tail 200
```

## 8. 部署后验收

必须至少验证：

```bash
curl -k https://你的域名/healthz
curl -k https://你的域名/readyz
```

期望：

1. `healthz` 返回可用
2. `readyz` 返回 `hard_gate_passed=true`

再做浏览器侧验收：

1. 打开 `/`
2. 打开 `/mobile`
3. 打开 `/review/factory`
4. 管理员登录
5. 主操或机台入口登录
6. 提交一条测试数据并确认审阅端可见

## 9. 现场首轮试跑建议

建议分三步：

1. `Day 1`：1 个管理员 + 1 个车间的主操/机台账号 + 1-2 个专项 owner 内部冒烟
2. `Day 2`：选 1 个车间单班次真实试点
3. `Day 3`：扩到 2-3 个岗位并开启管理端日审阅

Phase 1 首轮试跑不建议同时做：

1. MES 联调
2. 企业微信正式消息大规模推送
3. 成本核算 / 排产 / 合同排序自动化

本轮快速试跑额外建议：

1. GitHub 远端先接入，云主机可直接拉取代码
2. 先放一个车间，不直接全厂放量
3. 企业微信正式入口先不接，优先浏览器 / 钉钉链路

## 10. 降级与回滚

若上线后仅需暂停消息与发布，可修改：

```env
AUTO_PUSH_ENABLED=false
AUTO_PUBLISH_ENABLED=false
```

若需整体回退到上一版：

1. 保留当前数据库卷
2. 切回上一版代码
3. 重新执行：

```bash
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d --build
```

若数据库结构已变更，必须先确认 Alembic 迁移是否可回退，不能直接假设可逆。

## 11. 当前阻塞项判断

从当前仓库状态看，真正的上线阻塞主要是：

1. 发布基线尚未冻结，工作区仍有较多未提交改动
2. 正式域名、证书和生产 `.env` 尚未确认
3. 正式身份入口配置与账号映射尚未最终确认；若本轮只走浏览器 / 钉钉，企业微信可信域名/回调地址不构成当前阻塞
4. 现场主数据和试点账号尚需最终过一遍
5. 还需要一轮真实 UAT，而不只是本地技术验证

## 12. 当前建议结论

按现在的技术完成度：

1. `2-4 天` 可上云试跑
2. `5-7 天` 可进入内部正式试点
3. `10-15 天` 可达到较稳的现场落地状态

若企业微信正式联通、域名备案、现场培训一起算，建议按 `2-3 周` 管理预期更稳。
