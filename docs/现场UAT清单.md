# 现场 UAT 清单

## 目标

验证当前 `Phase 1` 是否已经满足：

`岗位直录 -> 系统自动处理 -> 管理端一目了然`

本轮默认是 `一个车间快速试跑`，企业微信正式入口先不接，先走浏览器访问链路。

## 参与角色

1. 1 个管理员
2. 1 个车间的主操或机台账号 1 人
3. 该车间专项 owner 1-2 人
   - 建议优先 `成品库 / 水电气 / 计划科`

## 测试场景

### 场景 1：主操正常填报

步骤：

1. 主操进入 `/entry`
2. 进入填报页
3. 录入一条正常原始值
4. 提交

预期：

1. 页面能进入，不迷路
2. 字段能理解，不需要额外解释
3. 提交成功后系统自动处理
4. 审阅端能看到对应来源和结果卡变化

### 场景 2：主操继续填下一卷

步骤：

1. 主操完成一条正常提交
2. 直接继续下一卷

预期：

1. 不需要重新选班次和机台
2. 连续录入提示不干扰，但能帮上手
3. 下一卷路径顺畅

### 场景 3：专项 owner 补录

步骤：

1. 成品库 / 水电气 / 计划科账号进入 `/entry`
2. 完成一条 owner-only 补录

预期：

1. 页面标题和字段分组贴合岗位
2. 不出现随行卡主操语义
3. 审阅端能显示专项 owner 数据来源或对应汇总结果

### 场景 4：管理员审阅

步骤：

1. 管理员进入 `/manage/factory`
2. 查看 `今日摘要 / 今日上报状态 / 交付与闭环 / 来源泳道`
3. 进入 `/manage/workshop`

预期：

1. 能一眼看懂数据来自哪里
2. 能看懂当前处理到了哪一步
3. 能一眼看出缺报、异常、退回、交付风险
4. 不暴露内部字段值如 `owner_only`

### 场景 5：权限边界

步骤：

1. 主操账号登录
2. 尝试切换不属于自己的数据
3. 管理员登录后查看全局

预期：

1. 主操只能看到自己的填报面
2. 管理员能看到审阅面
3. 不出现串看、串改

## 记录方式

每个场景记录：

1. 是否通过
2. 发现的问题
3. 是否阻断上线
4. 是否需要现场培训而不是改代码

## 阻断级问题定义

以下任一出现，都不应直接上正式试点：

1. 主操无法进入或无法提交
2. 专项 owner 无法补录
3. 审阅端看不到来源与状态
4. 权限串看或串改
5. `readyz` 不通过
