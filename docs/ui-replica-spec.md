# UI 复刻规范（参考图级生产指挥中心执行规格）

本规格用于持续约束 aluminum-bypass 的前端、路由、权限、后端聚合接口和视觉审计。目标不是做静态复刻，而是把参考图的页面颗粒度、业务分工、卡片密度和交互层级落成可运行的生产协同系统。

核心原则：

- 视觉贴近参考图：浅蓝灰背景、白底细边框卡片、蓝色编号、高密 KPI、紧凑表格、业务图形。
- 功能贴近参考图：每个编号模块都有真实数据、状态判断和可执行动作。
- 三端分离：录入端、审阅端、管理端各自只做本端任务，不交叉暴露导航。
- 兼容旧入口：旧 URL 和 route name 保持兼容，允许重定向和降权，不直接删除。
- 不使用英文小字副标题，标题统一为“编号 + 中文中心名”。
- 高科技感来自数据流、状态流、产线图形、精确网格和 AI 结果动效，不靠无意义背景装饰。

## 三端执行边界

录入端只负责录入。

- 允许：今日班次、待填任务、快速填报、高级填报、OCR、草稿、历史、异常补录、提交反馈、录入端智能提示。
- 禁止：厂级看板、车间看板、审阅任务、成本中心、AI 总控工作台、运维中心、权限治理、主数据配置。
- 路由：`/entry`、`/entry/report/*`、`/entry/advanced/*`、`/entry/ocr/*`、`/entry/attendance`、`/entry/history`、`/entry/drafts`。
- 权限：fill-only 用户不得看到审阅端或管理端导航，访问 `/review/*` 或 `/admin/*` 必须回跳 `/entry`。

审阅端只负责看数、审阅、处置和交付。

- 允许：总览、厂级/车间看板、审阅任务、日报交付、质量告警、差异治理、成本解释、AI 生产摘要。
- 禁止：用户管理、角色权限配置、主数据维护、模板字段配置、系统发布回滚配置。
- 路由：`/review/overview`、`/review/factory`、`/review/workshop`、`/review/tasks`、`/review/reports`、`/review/quality`、`/review/reconciliation`、`/review/cost-accounting`、`/review/brain`。

管理端只负责配置、治理、主数据和运维。

- 允许：数据接入、字段映射、导入历史、健康探针、AI live probe、权限治理、审计日志、主数据、模板字段。
- 禁止：现场填报、日常审阅任务处理、填报表单提交。
- 路由：`/admin`、`/admin/ingestion`、`/admin/field-mapping`、`/admin/ops`、`/admin/governance`、`/admin/master`、`/admin/master/templates`、`/admin/users`。

兼容规则：

- `/mobile/*` 重定向到 `/entry/*`。
- `/dashboard/*` 重定向到 `/review/*`。
- `/master/*` 重定向或兼容到 `/admin/*`。
- 旧 URL 和 route name 保持兼容，不破坏 `router.push({ name })`、二维码入口、E2E 和上线闸门。

## 页面构建规则

中心页不得只放静态卡片。每个中心页至少接入一个真实数据来源、一个状态判断和一个可执行动作或跳转。

页面结构：

- 编号标题区：例如 `01 系统总览主视图`、`07 审阅中心`。
- KPI 摘要区：3-7 个高密 KPI，含单位、趋势、状态。
- 主表格或主图形区：表格、产线示意、流程图、成本堆叠、趋势图。
- 摘要/风险/趋势区：Top 风险、异常、AI 建议、交付状态或运维状态。
- 固定操作区：表格上方工具条或卡片底部按钮，不能散落。

文案规则：

- 清理旧前端残余设计中的解释性长文案、软萌文案、演示稿文案和重复提示。
- 必要说明转成字段、状态标签、步骤标题、操作按钮或数据缺口提示。
- 不使用英文小字副标题。
- 不用长段文字撑版面。

## 参考图组件契约

前端组件层必须围绕参考图结构组织，避免在旧页面上零散补样式。

- `ReferencePageFrame`：统一中心页外框，负责编号标题、顶部工具区、主网格。
- `ReferenceModuleCard`：白底、细边框、紧凑间距、右上动作位。
- `ReferenceKpiTile`：小图标圆片、大数字、单位、趋势、状态。
- `ReferenceStatusTag`：红/绿/蓝/橙紧凑状态标签。
- `ReferenceDataTable`：紧凑表格，统一表头、行高、状态列。
- `ReferenceFlowGraphic`：产线示意、流程连接线、数据流动线。

组件 props 禁止使用以下解释性命名：`description`、`explanation`、`helperText`、`tooltip`、`note`、`rationale`。需要说明时，应改成业务字段、状态标签、操作入口或错误状态。

## 视觉与动效规则

视觉基线：

- 全局背景使用极浅蓝灰。
- 白底卡片、1px 蓝灰边框、12-16px 圆角、轻阴影。
- 表格行高紧凑，表头浅蓝灰，状态 tag 小而清楚。
- 图标使用蓝色线性或圆片风格，不能混用大色块插画。
- 装饰图必须是业务图形：产线、数据流、工序流、设备轮廓、状态雷达。

动效规则：

- 允许模块 stagger 入场、KPI 数字轻量滚动、产线流向线缓慢流动、异常点轻脉冲、AI 卡片结果出现动画。
- 禁止大面积持续闪烁、干扰表单输入的动画、移动端强动画、影响性能的常驻 canvas 背景。

## 后端适配与聚合接口契约

新增聚合接口服务参考图模块，不替代旧接口。旧 `/api/v1/*` 核心接口继续保留。

建议聚合接口：

- `GET /api/v1/command/overview`
- `GET /api/v1/command/review-center`
- `GET /api/v1/command/entry-terminal`
- `GET /api/v1/command/cost-center`
- `GET /api/v1/admin/ops-overview`
- `GET /api/v1/admin/governance-overview`
- `GET /api/v1/admin/master-overview`

聚合 view model 字段：

- `module_id`：对应 01-15 模块编号。
- `title`：中文中心标题。
- `kpis`：KPI 数组。
- `status_summary`：状态摘要。
- `primary_rows`：主表格行。
- `trend_series`：趋势或图形数据。
- `actions`：可执行动作。
- `freshness` 或 `updated_at`：数据时效。

KPI 字段：

- `label`
- `value`
- `unit`
- `trend`
- `status`
- `icon_key`

三端裁剪：

- entry surface 只返回填报任务和录入辅助信息。
- review surface 返回审阅、风险、交付、成本、AI 摘要。
- admin surface 返回配置、治理、运维、主数据。
- 同一用户拥有多端权限时，仍按当前访问 surface 裁剪响应。

高科技感数据基础：

- 产线图需要节点、连接线、状态和当前流转阶段。
- 风险脉冲需要风险等级、风险原因和处置状态。
- KPI 动画需要上一值、当前值和趋势方向。
- 成本图需要分项、占比、趋势和归因。
- AI 卡片需要摘要、建议、证据来源和 mock/live 标记。

## 响应式规则

全局响应式替代“移动端预览模块”。

- `>= 1280px`：参考图高密桌面布局。
- `900-1279px`：两列/三列卡片布局，侧栏可收缩。
- `< 900px`：三端进入移动布局，Entry 端优先；Review/Admin 只保留摘要、Top 风险和关键动作。

一致性要求：

- 桌面、平板、手机使用同一 token、同一状态色、同一编号体系。
- 页面宽度变化时允许列数变化、侧栏折叠、表格转卡片，不允许视觉风格漂移。
- 核心模块顺序保持稳定，避免换设备后找不到功能。

## 视觉审计清单

每轮 UI 改动后按以下颗粒度对照参考图：

- 模块编号是否正确。
- 中文标题是否准确，是否无英文副标题。
- 卡片比例是否接近参考图。
- KPI 数量、数字层级、单位和趋势是否清晰。
- 主表格/主图形是否占据页面核心区域。
- 右侧摘要/风险/趋势区是否存在。
- 操作区是否固定在工具栏或卡片底部。
- 状态色是否表达业务状态。
- 录入端是否没有审阅/管理入口。
- 审阅端是否没有主数据/配置入口。
- 管理端是否没有现场填报入口。
- 窄屏是否只重排，不改变视觉语言。

视觉审计脚本至少覆盖：

- `/login`
- `/entry`
- `/review/overview`
- `/review/tasks`
- `/review/cost-accounting`
- `/admin`
- `/admin/ops`

## TDD 与验收门禁

所有新增页面、权限边界、聚合接口和规格升级采用 TDD 红绿循环。

流程：

1. 先写契约测试或 E2E，锁定期望行为。
2. 运行测试确认红灯，失败原因必须是能力缺失或规格缺失。
3. 做最小实现让测试变绿。
4. 必要时重构，但保持测试绿。
5. 更新视觉审计清单或规格文档，避免口头约束。

完成前必须至少运行：

- `python -m pytest backend/tests -q`
- `npm --prefix frontend run e2e`
- `npm --prefix frontend run build`

如果本轮只改文档或测试，可运行相关测试并说明未运行 UI 构建的原因；涉及前端代码时必须跑构建，涉及主链路时必须跑 E2E。
